### 1.3 Asymmetry policy (drives TM-LEAK.3)

A WebSocket security guard that fires on one route but not the other is a leak. The asymmetric-guard sweep (`TM-LEAK.3`) enforces — by mechanical script, not by hand-audit — that **for each guard category**, both `/api/wrapper` and `/api/phone/:token` either:

- both apply the guard with identical parameters, OR
- one route explicitly opts out with a rationale comment on the same line as the absence, anchored to a `TM-CAT.N` ID.

The categories and the policy per category:

| Guard category                  | wrapper_ws.rs               | phone_ws.rs                 | Asymmetry rationale (if any)                                                       |
|---------------------------------|-----------------------------|-----------------------------|------------------------------------------------------------------------------------|
| `Origin` match if configured    | required                    | required                    | none — both symmetric (TM-WS.1, .2)                                                |
| `Origin` fail-closed on missing | NOT enforced                | enforced                    | wrapper is CLI; browsers never reach it (TM-WS.3 + spec §1.1 rationale)            |
| `max_message_size = 64 KB`      | required                    | required                    | none (TM-WS.4)                                                                     |
| `max_frame_size = 64 KB`        | required                    | required                    | none (TM-WS.5)                                                                     |
| `HELLO_TIMEOUT = 10 s`          | required                    | NOT applicable              | phone WS has no post-upgrade hello (different protocol shape) — comment in phone_ws.rs anchors this asymmetry to TM-RATE.8 |
| 30 s ping keepalive             | required                    | required                    | none (TM-WS.6)                                                                     |
| `PONG_DEADLINE = 90 s`          | required                    | required                    | none (TM-RATE.7 / TM-WS.7)                                                         |
| `SINK_SEND_TIMEOUT = 5 s`       | required                    | required                    | none (TM-RATE.6)                                                                   |
| Per-conn `ConnRateLimiter`      | required                    | required                    | different caps per direction; same primitive (TM-RATE.3)                            |
| `ConnectInfo<SocketAddr>`       | required                    | required                    | none (4.2 logging, 4.6 rate limit)                                                  |
| Token-length strict equality    | NOT applicable              | required                    | wrapper has no path token; phone has it (TM-WS.11)                                  |
| `permessage-deflate` negotiated | refused                     | refused                     | none (TM-WS.8) — both must stay off                                                 |
| Sec-WebSocket-Protocol honored  | refused                     | refused                     | none (TM-WS.12) — neither route calls `.protocols(...)`                             |

The sweep script (`§3.6` below) parses these expectations and asserts the source files match.

## 2. Detailed mitigation design

### 2.1 TM-WS.3 — fail-closed missing Origin (phone_ws only)

**Today** (`phone_ws.rs:42-51`):

```rust
if let Some(expected) = state.public_origin.as_deref() {
    if let Some(origin) = headers.get(header::ORIGIN).and_then(|v| v.to_str().ok()) {
        if origin != expected {
            return StatusCode::FORBIDDEN.into_response();
        }
    }
}
```

The inner `if let Some(origin)` doesn't enter when `Origin` is absent → upgrade proceeds. Browsers always send `Origin`, but a malicious script running inside a victim's browser could **strip** it by using a `<form>` POST or a deliberately-crafted request from a stale tab. More directly: any non-browser CSWSH-style attacker just doesn't send the header.

**Change:** when `public_origin` is set, **require** the `Origin` header on phone_ws and reject 403 if absent.

```rust
// TM-WS.3 — when public_origin is configured, the Origin header is
// REQUIRED on phone_ws. Phone WS is browser-served (the page at /s/:token
// opens a same-origin WebSocket), so a legitimate browser will always
// send Origin. A missing Origin is therefore either (a) a non-browser
// client probing the endpoint with a stolen token, or (b) a browser
// quirk we don't support. Both deserve 403.
if let Some(expected) = state.public_origin.as_deref() {
    let origin = headers
        .get(header::ORIGIN)
        .and_then(|v| v.to_str().ok());
    match origin {
        Some(o) if o == expected => {}
        _ => return StatusCode::FORBIDDEN.into_response(),
    }
}
```

**Wrapper_ws stays permissive** — wrappers are CLI processes, never browsers; demanding `Origin` would break legitimate `claude-phone-wrapper` connections. The asymmetry is **deliberate and documented** with the comment + the asymmetric-guard policy table at §1.3 above.

### 2.2 TM-WS.9 — fail-loud at boot if production and public_origin missing

**Today:** the gateway boots fine with `public_origin = None` in any environment. In production this disables the `TM-WS.1/.2/.3` Origin defense at a stroke, silently.

**Change:** add an `environment` field to `GatewayConfig`:

```rust
// crates/claude-phone-gateway/src/config.rs

#[derive(Debug, Clone, Copy, Deserialize, Default)]
#[serde(rename_all = "snake_case")]
pub enum Environment {
    #[default]
    Development,
    Production,
}

#[derive(Debug, Clone, Deserialize)]
pub struct GatewayConfig {
    // ... existing fields ...
    #[serde(default)]
    pub environment: Environment,
    #[serde(default)]
    pub public_origin: Option<String>,
}

impl GatewayConfig {
    pub fn load(path: &std::path::Path) -> anyhow::Result<Self> {
        let raw = std::fs::read_to_string(path)?;
        let cfg: Self = toml::from_str(&raw)?;
        cfg.validate()?;
        Ok(cfg)
    }

    fn validate(&self) -> anyhow::Result<()> {
        // TM-WS.9 — production must declare its public origin so that the
        // Origin defense (TM-WS.1, .2, .3) actually fires. A misconfigured
        // production gateway is detected at startup rather than after the
        // first malicious request.
        if matches!(self.environment, Environment::Production)
            && self.public_origin.is_none()
        {
            anyhow::bail!(
                "production environment requires public_origin to be set \
                 in gateway.toml (TM-WS.9). Refusing to start."
            );
        }
        Ok(())
    }
}
```

**Default is `Development`** so existing dev/test configs continue to boot. Production must declare `environment = "production"` explicitly in `/etc/claude-phone/gateway.toml`.

**Deploy-side anchor:** `deploy/scripts/deploy.sh` is updated to emit a warning if `environment` is unset in the production config (it can't *force* a value — operator-managed file — but it can tell the operator). Same one-line `grep -q '^environment' "$ETC_DIR/gateway.toml" || echo "WARN: ..."` style used in 4.7's `verify_perms` post-step.

### 2.3 TM-WS.8 — permessage-deflate compression negotiation OFF

**Today:** axum's `WebSocketUpgrade` does not negotiate `permessage-deflate` because we never call `.with_compression()` or similar. The mitigation is therefore **already in effect** — we just don't have a test pinning it.

**Change:** add a forward-looking negative-assertion test that connects to both WS routes with `Sec-WebSocket-Extensions: permessage-deflate` and asserts the server's 101 response does **not** include `Sec-WebSocket-Extensions` in its headers (or, if it does, the value is empty / does not contain `permessage-deflate`).

Why this matters as a forward-looking test:

- Compression on tiny PTY frames + attacker-controlled content is a **CRIME / BREAST** style oracle. We never want it on.
- The default is correct *today*, but any future contributor calling `WebSocketUpgrade::with_compression(true)` (if axum ever adds such a method) would silently re-open the hole. A failing CI test catches that.

No code change in this commit beyond the test itself.

### 2.4 TM-WS.12 — Sec-WebSocket-Protocol negotiation strict-unset

**Today:** neither WS route calls `WebSocketUpgrade::protocols(...)`. axum's default behavior is to ignore any client-offered `Sec-WebSocket-Protocol` and respond without the header. Mitigation is **already in effect**, no test today.

**Change:** add a forward-looking negative-assertion test that connects with `Sec-WebSocket-Protocol: claude-phone-v0, chat-v1` and asserts the server's 101 response does **not** include `Sec-WebSocket-Protocol`.

Why this matters:

- Subprotocol negotiation lets an attacker probe for behavior differences across versions. If we ever *do* version the protocol via subprotocols, the negotiation logic must be explicit, audited, and strict-match — never "first acceptable". This test pins the current "no negotiation" baseline until that day arrives.

### 2.5 TM-WS.7 — regression-verify (4.6 implements)

4.6 lands the `PONG_DEADLINE = 90 s` watchdog in both `phone_ws.rs` and `wrapper_ws.rs` with the same code shape (`AtomicU64` last-pong, keepalive tick checks elapsed, closes the socket on overrun).

This sub-spec adds an **asymmetry-pin test**: connect both routes, ignore Pings on both, assert both close within `PONG_DEADLINE + ping_interval + 5 s` jitter. If a future refactor splits the implementation and one side regresses, this test fires before 4.6's per-route test does (because it asserts on *equality of behavior*, not just on the absolute deadline).

Implementation lives in `tests/websocket.rs` (new test module, 4.13 owns).

### 2.6 TM-WS.10 — regression-verify (4.6 implements)

Same shape as §2.5. 4.6 lands `header_read_timeout(Some(10 s))` on the hyper builder. 4.13 adds an asymmetry-pin test: open two TCP sockets to the gateway, dribble headers on both, assert both close within 11 s. Both because there's only one listener, but the test exercises both URL paths to assert the per-route handler invocation reaches the timeout the same way.

### 2.7 TM-LEAK.3 — asymmetric-guard sweep

A Bash script that grep-asserts the asymmetry policy table at §1.3.

```bash
#!/usr/bin/env bash
# scripts/asymmetric_guards.sh
#
# TM-LEAK.3 — assert WS route guards are either symmetric across
# /api/wrapper and /api/phone/:token, or explicitly opted out with a
# rationale comment anchored to a TM-CAT.N ID.
#
# Exit 1 on any expected guard missing or any unexpected guard present.
# Intended to run in pre-commit and CI; <100 ms on the current source tree.

set -euo pipefail
cd "$(dirname "$0")/.."

WRAPPER="crates/claude-phone-gateway/src/routes/wrapper_ws.rs"
PHONE="crates/claude-phone-gateway/src/routes/phone_ws.rs"

# --- Guard category presence: both routes ---------------------------------

assert_in_both() {
    local pat="$1"; local why="$2"
    grep -qE "$pat" "$WRAPPER" || { echo "MISSING in $WRAPPER: $pat ($why)"; exit 1; }
    grep -qE "$pat" "$PHONE"   || { echo "MISSING in $PHONE: $pat ($why)";   exit 1; }
}

assert_in_both 'max_message_size\(MAX_WS_MESSAGE_BYTES\)' 'TM-WS.4'
assert_in_both 'max_frame_size\(MAX_WS_MESSAGE_BYTES\)'   'TM-WS.5'
assert_in_both 'MAX_WS_MESSAGE_BYTES: usize = 64 \* 1024' 'TM-WS.4/5'
assert_in_both 'SINK_SEND_TIMEOUT'                        'TM-RATE.6'
assert_in_both 'PONG_DEADLINE'                            'TM-RATE.7 / TM-WS.7'
assert_in_both 'ConnRateLimiter::new'                     'TM-RATE.3'
assert_in_both 'ConnectInfo<SocketAddr>'                  '4.2 logging / 4.6 rate'
assert_in_both 'tokio::time::interval\(std::time::Duration::from_secs\(30\)\)' 'TM-WS.6'
assert_in_both 'public_origin.as_deref'                   'TM-WS.1, .2 Origin check'

# --- Asymmetric (one route only) ------------------------------------------

# HELLO_TIMEOUT is wrapper-only by design (phone has no post-upgrade hello).
grep -q 'HELLO_TIMEOUT' "$WRAPPER" \
    || { echo "MISSING HELLO_TIMEOUT in $WRAPPER (TM-RATE.8)"; exit 1; }
if grep -q 'HELLO_TIMEOUT' "$PHONE"; then
    echo "UNEXPECTED HELLO_TIMEOUT in $PHONE — phone has no post-upgrade hello"
    exit 1
fi

# Token-length strict check is phone-only (wrapper has no path token).
grep -q 'token_str.len() != SessionToken::LEN' "$PHONE" \
    || { echo "MISSING token length check in $PHONE (TM-WS.11)"; exit 1; }
if grep -q 'token_str.len() != SessionToken::LEN' "$WRAPPER"; then
    echo "UNEXPECTED token length check in $WRAPPER — wrapper has no token in path"
    exit 1
fi

# Fail-closed-on-missing-Origin is phone-only (TM-WS.3 wrapper rationale at §2.1).
grep -qE 'Some\(o\) if o == expected' "$PHONE" \
    || { echo "MISSING fail-closed missing-Origin guard in $PHONE (TM-WS.3)"; exit 1; }
if grep -qE 'Some\(o\) if o == expected' "$WRAPPER"; then
    echo "UNEXPECTED fail-closed missing-Origin guard in $WRAPPER — wrapper is CLI"
    exit 1
fi

# --- Forbidden: any WebSocketUpgrade method we don't want -----------------

if grep -qE 'with_compression|\.protocols\(' "$WRAPPER" "$PHONE"; then
    echo "FORBIDDEN: WS compression or subprotocol negotiation must remain off (TM-WS.8, .12)"
    exit 1
fi

echo "asymmetric guards: OK"
```

Made executable in the same commit; hooked into `scripts/security_invariants.sh` (from 4.7) as one of its checks; added as a step in the CI `security-checks` job introduced by 4.7.

The script is **intentionally simple grep**, not AST-based. The patterns are stable strings that grep cleanly; any future refactor that renames `PONG_DEADLINE` etc. will trip the sweep and force the rename to also update this script — which is the point: rename-and-forget is exactly the failure mode TM-LEAK.3 catches.

## 6. Risk-of-regression analysis

| Change                                                | Risk                                                                                                                  | Guard                                                                                                                                |
|-------------------------------------------------------|-----------------------------------------------------------------------------------------------------------------------|--------------------------------------------------------------------------------------------------------------------------------------|
| Fail-closed missing Origin on phone_ws                | Legitimate non-browser phone clients (e.g. tester scripts) without `Origin` get 403.                                  | Dev configs typically omit `public_origin` → check is skipped, no impact. Tests cover both branches. Documented behavior change; tester scripts already need to handle the existing `Origin` check on the value, so adding "must be present" is incremental. |
| Asymmetric guard policy formalized                    | Future contributors may add a guard to one route and forget the other; sweep script catches.                          | `asymmetric_guards.sh` runs in pre-commit and CI; reviewer also has the table at §1.3 to consult.                                    |
| `Environment` field added to config                   | Existing prod gateway.toml without the field defaults to `Development` and silently disables TM-WS.9 fail-loud.       | Default is **safe** (Development always works). The risk is only that prod operators forget to add `environment = "production"`; mitigated by the deploy.sh warning (§2.2) and the SECURITY.md operator runbook from 4.7. |
| Compression / subprotocol negative-assertion tests    | Tests fail if a future axum upgrade changes default behavior to negotiate something automatically.                    | Failure mode is **loud and correct** — we want to be notified if axum starts negotiating compression by default. The test is the canary. |
| Asymmetric-guard sweep grep-based                     | Refactor that renames `PONG_DEADLINE` etc. trips the sweep until the script is updated.                               | Intentional. Refactor that renames a security-relevant constant must also update the audit harness — that's the design.              |
