## 1. Scope

This sub-spec covers four `TM-AUTH.*` items. Items already GREEN in the
2026-05-23 sweep (`TM-AUTH.1, .2, .5, .8, .9, .10, .11`) are not touched
here — they are foundation and the regression sweep (master §7) guarantees
they stay.

| ID         | Mitigation                                                                                  | Pre-spec status | Post-spec status |
|------------|---------------------------------------------------------------------------------------------|-----------------|------------------|
| TM-AUTH.3  | Single-phone-per-session: second phone refused (`SessionTaken`) while first attached         | PARTIAL (behaviour at `registry.rs:78-84`, no forward-looking test) | GREEN |
| TM-AUTH.4  | Revocation via `/etc/claude-phone/revoked.toml` + SIGHUP reload (lock-free hot swap)         | TODO            | GREEN            |
| TM-AUTH.6  | 90-day API key rotation cadence (runbook entry, no code)                                    | TODO            | GREEN            |
| TM-AUTH.7  | Structured auth-failure log with correlation ID, **no secret payload**                       | TODO            | GREEN            |

### 1.1 Out of scope here

- TM-AUTH.1/.2/.5/.8/.9/.10/.11 — already GREEN, regression-tested by sweep.
- TM-RATE.2 (auth-attempt rate limit) — sub-spec 4.6 owns this; it consumes
  the `TM-AUTH.7` audit log via a tracing layer reader.
- TM-INFRA.4 (`fail2ban` jail) — sub-spec 4.9 owns this; it watches the
  same audit log.

### 1.2 Open-question resolutions adopted

None affecting this sub-spec. (OQ-7/OQ-8 cover wrapper config file perms
and gitleaks tooling — see sub-spec 4.7.)

### 1.3 Design choice that supersedes the threat-model draft

The original threat model §6.1 F2 flagged "kicks previous" as the default
single-phone semantic. That has been corrected (TM edit in this same change
set) to "refuses second". Rationale: kick-previous lets an attacker holding
a stolen token forcibly disconnect the legit user and take over the
session. Refuse-second locks the attacker out as long as the user holds
the connection — strictly safer. Existing code at `registry.rs:78-84`
already implements refuse-second; this sub-spec hardens it with a
forward-looking test and an inline `// TM-AUTH.3:` comment so a future
refactor cannot quietly invert it.

## 2. Dependencies introduced

| Crate         | Version          | Why                                                                                                                                                                       | In-house alternative considered                                                                                                                                                                  |
|---------------|------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| `uuid`        | `1`, `features = ["v7"]` | Correlation ID for the auth-failure audit log (TM-AUTH.7). v7 is timestamp-prefixed, monotonically sortable, and 128-bit random tail. ~5 KLOC, used by tokio ecosystem already (likely transitive). | Roll a `[u8; 16]` of `Instant::now().elapsed().as_nanos()` ++ `OsRng` 8 bytes. Loses RFC4122 compatibility for log greppers and external tooling (`fail2ban`, log shippers).                       |
| `arc-swap`    | `1`              | Lock-free hot-swap of `allowed_keys` and `revoked` on SIGHUP (TM-AUTH.4). The auth hot path (`verify_api_key`) reads on every WrapperHello; a `RwLock` read-lock on every call adds contention, and an `Arc<Mutex<Arc<...>>>` is the same. arc-swap is purpose-built. | `Arc<Mutex<Arc<Vec<ApiKey>>>>` — works but takes a mutex on every auth check. Or `unsafe AtomicPtr<Arc<...>>` — manual drop-ordering, easy to mis-design. Both worse than the 1-line crate alternative.   |

### 3.1 `crates/claude-phone-gateway/src/config.rs`

Add `RevokedConfig` struct loaded from a separate file:

```rust
/// Lists of credentials that, even if present in the legitimate allowlist,
/// MUST be rejected. Loaded from `/etc/claude-phone/revoked.toml` (path is
/// taken from `--revoked` CLI arg, defaults to a sibling of the main
/// config). Empty file (or missing-file) means "no revocations".
///
/// Hot-swappable via SIGHUP — see `main.rs::reload_signal_task`.
#[derive(Debug, Default, Clone, Deserialize)]
pub struct RevokedConfig {
    #[serde(default)]
    pub api_keys: Vec<ApiKey>,
    #[serde(default)]
    pub session_tokens: Vec<SessionToken>,
}

impl RevokedConfig {
    pub fn load(path: &std::path::Path) -> anyhow::Result<Self> {
        if !path.exists() {
            return Ok(Self::default());
        }
        let raw = std::fs::read_to_string(path)?;
        Ok(toml::from_str(&raw)?)
    }
}
```

Also add a method to `GatewayConfig` to discover the revoked-file path
(if the user wants to keep it next to `gateway.toml`):

```rust
pub fn default_revoked_path(config_path: &std::path::Path) -> std::path::PathBuf {
    config_path
        .parent()
        .unwrap_or_else(|| std::path::Path::new("."))
        .join("revoked.toml")
}
```

Lines roughly added: ~30 (currently file is 67 lines).

### 3.2 `crates/claude-phone-gateway/src/auth.rs` (rename current 7-line file)

Replace with a small module holding shared auth state and helpers. **No
breaking change to the API surface visible from routes** — keep
`verify_api_key` callable, just rebind to a new helper.

```rust
use std::sync::Arc;

use arc_swap::ArcSwap;
use claude_phone_shared::{ApiKey, SessionToken};

use crate::config::RevokedConfig;

/// Hot-swappable auth state. Both fields are `ArcSwap<Vec<...>>` so the
/// SIGHUP handler can publish a new revocation list atomically without
/// taking a lock on the hot path.
#[derive(Clone)]
pub struct AuthState {
    pub allowed: Arc<ArcSwap<Vec<ApiKey>>>,
    pub revoked: Arc<ArcSwap<RevokedConfig>>,
}

impl AuthState {
    pub fn new(allowed: Vec<ApiKey>, revoked: RevokedConfig) -> Self {
        Self {
            allowed: Arc::new(ArcSwap::from_pointee(allowed)),
            revoked: Arc::new(ArcSwap::from_pointee(revoked)),
        }
    }

    /// True iff `provided` is in the allowlist AND not in the revoked list.
    /// Both checks use `subtle::ConstantTimeEq` via `ApiKey::ct_eq`.
    pub fn verify_api_key(&self, provided: &ApiKey) -> bool {
        // TM-AUTH.4: revocation check first — a revoked key must lose
        // even if it is still in the allowlist file (eventual consistency
        // window between revoking and removing from the allowlist).
        let revoked = self.revoked.load();
        if revoked.api_keys.iter().any(|r| r.ct_eq(provided)) {
            return false;
        }
        let allowed = self.allowed.load();
        // TM-AUTH.2: constant-time scan over allowlist (preserved from
        // the previous implementation).
        allowed.iter().any(|a| a.ct_eq(provided))
    }

    /// True iff `token` has been revoked. Phone attach path consults this
    /// before calling into the session registry.
    pub fn is_session_token_revoked(&self, token: &SessionToken) -> bool {
        // TM-AUTH.4: even if the session is still live in the registry,
        // a revoked token must be denied.
        let revoked = self.revoked.load();
        revoked.session_tokens.iter().any(|r| r.ct_eq(token))
    }
}

// TM-AUTH.2: kept for any caller that still wants the stateless variant.
// Production code in routes should construct `AuthState` once and use it.
pub fn verify_api_key(provided: &ApiKey, allowed: &[ApiKey]) -> bool {
    allowed.iter().any(|a| a.ct_eq(provided))
}
```

Roughly 50 lines (was 7).

### 3.7 `crates/claude-phone-gateway/src/main.rs`

Add a SIGHUP signal handler that re-reads the revoked file and publishes
to the `AuthState::revoked` `ArcSwap`. Pass the `Arc<ArcSwap<...>>`
handle from `build_app` to the signal task.

Sketch:

```rust
async fn reload_signal_task(
    revoked_path: PathBuf,
    revoked_handle: Arc<ArcSwap<RevokedConfig>>,
) {
    #[cfg(unix)]
    {
        use tokio::signal::unix::{signal, SignalKind};
        let mut sighup = match signal(SignalKind::hangup()) {
            Ok(s) => s,
            Err(e) => {
                tracing::warn!(error = %e, "SIGHUP signal install failed; revocation reload disabled");
                return;
            }
        };
        loop {
            sighup.recv().await;
            match RevokedConfig::load(&revoked_path) {
                Ok(new) => {
                    let api_count = new.api_keys.len();
                    let tok_count = new.session_tokens.len();
                    revoked_handle.store(Arc::new(new));
                    // TM-AUTH.7: structured reload event — counts only,
                    // never the actual revoked values.
                    tracing::info!(
                        api_keys = api_count,
                        session_tokens = tok_count,
                        "revoked list reloaded"
                    );
                }
                Err(e) => {
                    tracing::warn!(error = %e, "revoked reload failed");
                }
            }
        }
    }
    #[cfg(not(unix))]
    {
        // Windows dev only — no signal handling. drop both args to satisfy borrow.
        let _ = (revoked_path, revoked_handle);
    }
}
```

`main()` spawns this task after `build_app` returns the handle. CLI gets
an additional `--revoked <PATH>` arg defaulting to
`default_revoked_path(&cli.config)`.

Roughly +50 lines added.

```markdown
## Rotating an API key

Default cadence: every 90 days. Quarterly calendar reminder OK for one-person
ops. To rotate without downtime:

1. Generate a fresh key:
   `openssl rand -base64 32 | tr -d '=' | tr '/+' '_-'`
2. Add it to `/etc/claude-phone/gateway.toml` `api_keys` (keep the old one).
3. SIGHUP the gateway:  `sudo systemctl reload claude-phone-gateway`
   (or `kill -HUP <pid>` if no reload= in unit yet — sub-spec 4.9 wires it).
4. Distribute the new key to the wrapper user out-of-band.
5. After 24 h (or confirmed all wrappers cut over), remove the old key:
   - Move it to `revoked.toml` `api_keys` for one rotation cycle (paranoia
     against lingering background reconnects);
   - SIGHUP again;
   - After another 24 h, drop from both files.

## Revoking a single API key or session token (incident)

`revoked.toml` is the live deny list. Add the offender's key/token, SIGHUP,
done — no service restart. The deny list is checked before the allowlist
in `auth.rs::verify_api_key`, so a revoked key cannot reauth even if it is
also still in the allowlist (eventual-consistency window).
```

This is the only TM-AUTH.6 deliverable. **No code.**

## 6. Risk-of-regression notes

### 6.1 Anything that touches `verify_api_key`

The original 7-line `auth.rs::verify_api_key(&ApiKey, &[ApiKey]) -> bool`
free function is **preserved**. Any code currently calling it (tests, any
future caller) still compiles. The new `AuthState::verify_api_key(&self,
&ApiKey)` is the production code path. Both internally use
`ApiKey::ct_eq` — no `==` regression.

### 6.2 `WrapperWsState::allowed_keys: Arc<Vec<ApiKey>>` → `auth: AuthState`

This is an API-surface change for tests that mock `WrapperWsState` directly.
The only such location in tree is `tests/e2e_test.rs` — it constructs the
state via `build_app`, not directly, so it is unaffected. New tests
construct `AuthState::new` explicitly.

### 6.3 SIGHUP semantics on Windows

This sub-spec is **Unix-only** for SIGHUP. The `#[cfg(unix)]` gate ensures
the gateway compiles on Windows for dev (you, the user, develop on
Windows). On Windows the gateway falls back to "revocation list loaded
at startup, no reload". Documented inline.

### 6.4 ArcSwap memory ordering

`ArcSwap::store` is `AcqRel`; `ArcSwap::load` is `Acquire`. The hot path
calls `.load()` and dereferences via `Guard`. No risk of torn read; the
Arc inside is fully published before the swap is visible. Test #5 stress-
tests this (rapid swap + read interleaving).

### 6.5 Tracing layer interaction

`TraceLayer::make_span_with` in `http.rs` already redacts paths
(`TM-SECRET.6`). Adding `tracing::warn!` calls **inside** route handlers
(not in middleware) produces sibling events, not nested in the redacted
span. Test #9/#10 verify the WARN body itself never contains the secret;
the surrounding span is path-redacted regardless.

### 6.6 Forward-looking guards (do NOT remove without re-design)

- `// TM-AUTH.3:` comment in `registry.rs:78-84`.
- `// TM-AUTH.4:` comment in `auth.rs::verify_api_key`.
- `// TM-AUTH.4:` comment in `phone_ws.rs` revocation check.
- Test #1/#2/#3/#4/#5/#7/#9/#10 are invariant tests, not behaviour tests.
