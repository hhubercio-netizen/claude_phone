  Rationale: the wrapper user is a developer; their `~/.config/claude-phone/`

### 1.3 Defense-in-depth posture

Three layers cover api-key leakage by accidental disclosure:

1. **At rest** — file perms (`TM-SECRET.1` for gateway, `TM-SECRET.10` for
   wrapper) refuse the load if the file is broader than user-only or
   user+group. Compromise requires either root on the box or co-resident
   privilege escalation.
2. **In code** — derived `Debug` on container types is heuristically
   grepped at CI (`TM-LEAK.2`); `tracing::` macros with secret-named
   fields are hard-banned at CI (`TM-LEAK.1`); the inner secret types
   already redact themselves (`TM-SECRET.2/.3`, GREEN).
3. **In git** — pre-commit gitleaks (`TM-SECRET.8`) blocks the commit;
   CI gitleaks job catches a `--no-verify` bypass; `.gitleaksignore`
   tracks the small handful of intentional placeholders.

A single-layer regression (e.g., a future contributor disables the
pre-commit hook locally) does not lose the secret — CI still catches it
before the push reaches `origin/main`. The error-chain opacity
(`TM-SECRET.11`) and the SRI / CDN assertion (`TM-SUPPLY.5`) are
adjacent: they prevent the gateway response body and the static HTML
from becoming leak channels.

## 2. Dependencies introduced

**None — Rust or npm.**

The Rust-side perms check uses `std::os::unix::fs::PermissionsExt` (stable
std on Unix). The wrapper crate already depends on `anyhow`; this sub-spec
reuses it. On Windows the entire perms-check block is `#[cfg(unix)]`-gated.

### 2.1 External tooling (not a code-level dep)

`gitleaks` is an external binary. Two install paths:

| Path                   | When                                                       | Verification                                                                                  |
|------------------------|------------------------------------------------------------|------------------------------------------------------------------------------------------------|
| Local dev environment  | `scripts/install-pre-commit.sh` checks for `gitleaks` on `$PATH`; if missing, prints download URL and exits non-zero. Does NOT auto-install. | Reproducible: dev follows the printed URL (`https://github.com/gitleaks/gitleaks/releases`). |
| CI                     | `gitleaks/gitleaks-action@v2` pinned by major version + commit SHA. | GitHub Action; SHA pin lives in `ci.yml`.                                                      |

**Alternatives considered:**

| Tool         | Pros                                                       | Cons                                                                                                 | Decision        |
|--------------|------------------------------------------------------------|-------------------------------------------------------------------------------------------------------|-----------------|
| `gitleaks`   | Single static binary; entropy + regex; well-maintained.    | Some false positives on test fixtures (mitigated by `.gitleaksignore`).                              | **Adopted (OQ-8).** |
| `trufflehog` | Heavier, deeper (live verification of cloud creds).        | Overkill; we have no AWS/GCP creds to verify; slower; Python runtime.                                | Rejected.       |
| `git-secrets`| AWS-flavored regex; bash-only.                             | Regex only; no entropy; AWS-centric patterns.                                                         | Rejected.       |
| In-house     | Zero dep.                                                  | Reinventing entropy detection is the kind of work that fails silently — exactly what we are guarding. | Rejected.       |

`.gitleaksignore` (committed) lists fingerprints of intentional placeholder
strings (e.g., `REPLACE_WITH_FRESHLY_GENERATED_BASE64URL_NO_PAD` in
`gateway-dev.toml`-shaped example output in docs). Format: `<commit_sha>:<file>:<line>:<rule_id>`
per gitleaks docs.

## 3. Detailed changes per file

### 3.1 `crates/claude-phone-wrapper/src/config.rs` (52 → ~95 lines)

Add a `check_perms` function and call it from `load`. Unix-only.

```rust
impl WrapperConfig {
    pub fn load(path: &Path) -> anyhow::Result<Self> {
        // TM-SECRET.10: refuse to load a config file readable by group
        // or world. The wrapper config holds a long-lived api_key; if
        // it is readable by anyone other than the owner we have already
        // lost. On Windows this check is a no-op (NTFS ACLs are not a
        // good fit for an exact-bits check, and the dev box is the
        // single-user case anyway).
        #[cfg(unix)]
        Self::check_perms(path)?;

        let raw = std::fs::read_to_string(path)?;
        let cfg: Self = toml::from_str(&raw)?;
        Ok(cfg)
    }

    #[cfg(unix)]
    fn check_perms(path: &Path) -> anyhow::Result<()> {
        use std::os::unix::fs::PermissionsExt;

        let meta = std::fs::metadata(path)?;
        let mode = meta.permissions().mode() & 0o777;
        // Accept 0600 (rw owner) or 0400 (r owner). Anything broader,
        // including 0640 / 0660 / 0644, rejects.
        if mode & 0o077 != 0 {
            anyhow::bail!(
                "wrapper config {} mode is {:#o}, must be 0600 or 0400 \
                 (run: chmod 0600 {})",
                path.display(),
                mode,
                path.display()
            );
        }
        Ok(())
    }
}
```

Lines added: ~30 (file grows from 52 to ~85).

**Backwards compatibility:** existing callers of `WrapperConfig::load` get
a new failure mode on Unix. The wrapper main path (`main.rs::WrapperConfig::load(&cli.config)`)
is the only production caller; on existing dev boxes the user runs
`chmod 0600` once. The fail-loud anyhow message says exactly what to do.

### 3.2 `crates/claude-phone-gateway/src/error.rs` (33 → ~50 lines)

The current handler already returns an opaque `"internal error"` body for
`Internal` and `Io` variants (lines 26-29) and static enum strings for
4xx. **No behaviour change is needed.** This commit adds a leading
module comment and a code-level fence so a future maintainer adding a
new variant cannot quietly leak inner detail.

```rust
//! HTTP error responses for the gateway.
//!
//! TM-SECRET.11: response bodies for **every** variant must stay static
//! or sanitized. Specifically:
//!
//! - 4xx variants return a fixed `enum.to_string()` (no captured data).
//! - 5xx variants return the literal `"internal error"` regardless of
//!   the inner anyhow / io chain — that chain only goes to tracing.
//!
//! When adding a new variant: if the variant carries dynamic data
//! (path, field name, deserialization error), the data MUST be logged
//! via `tracing::error!(error = ?self, …)` and the response body MUST
//! be a static string. The included forward-looking test
//! `gateway_response_bodies_never_contain_inner_detail` asserts this.
```

Plus inline `// TM-SECRET.11: …` at the 5xx arm:

```rust
Self::Internal(_) | Self::Io(_) => {
    tracing::error!(error = ?self, "internal error");
    // TM-SECRET.11: opaque body. Inner chain stays in the log; the
    // response body never reveals server-side state.
    (StatusCode::INTERNAL_SERVER_ERROR, "internal error".into())
}
```

Lines added: ~15 (file grows from 33 to ~50).

### 3.3 New file `crates/claude-phone-gateway/tests/error_response_test.rs` (~120 lines)

Three tests:

```rust
// Test 5: 500 body is opaque
#[tokio::test]
async fn gateway_500_body_does_not_contain_internal_detail() { … }

// Test 6: 4xx bodies are short static strings (no >80 chars, no typenames)
#[tokio::test]
async fn gateway_4xx_bodies_are_static_strings() { … }

// Test 7: malformed input → response body is terse, no serde verbose hint
#[tokio::test]
async fn gateway_malformed_json_response_body_is_terse() { … }
```

Each test fires a real request via `tower::ServiceExt::oneshot` against a
`build_app(…)` stack and asserts on the response body string. Forward-
looking: assertions are negative (`!body.contains("…")` for forbidden
substrings) rather than positive equality so adding a new variant with
a longer message stays compatible AS LONG AS it doesn't introduce a
forbidden substring.

Forbidden substrings (case-insensitive): `"stack"`, `"panicked"`,
`"BACKTRACE"`, `"thread '"`, `"src/"`, `"expected"`, `"deserialize"`,
`"::"` (Rust path separator — catches typenames), `"$crate"`,
`"line "` followed by digits, `"column "` followed by digits.

### 3.4 New file `scripts/install-pre-commit.sh` (~40 lines)

Installs the git hook. Idempotent. Does NOT auto-download gitleaks —
contributors run it once.

```bash
#!/usr/bin/env bash
# TM-SECRET.8: install pre-commit hook that runs gitleaks, fmt, clippy,
# and security_invariants.sh before every commit. Fails fast if any
# guard would catch a real issue.
set -euo pipefail

REPO_ROOT="$(git rev-parse --show-toplevel)"
HOOK_SRC="$REPO_ROOT/scripts/pre-commit.sh"
HOOK_DST="$REPO_ROOT/.git/hooks/pre-commit"

if ! command -v gitleaks >/dev/null 2>&1; then
    echo "gitleaks not on PATH."
    echo "Download from https://github.com/gitleaks/gitleaks/releases and add to PATH."
    echo "After installing, re-run this script."
    exit 1
fi

if ! command -v cargo >/dev/null 2>&1; then
    echo "cargo not on PATH (install Rust)." >&2
    exit 1
fi

install -m 0755 "$HOOK_SRC" "$HOOK_DST"
echo "Installed pre-commit hook at $HOOK_DST"
echo "It runs: cargo fmt --check, cargo clippy -D warnings, gitleaks, security_invariants.sh"
echo "To bypass (do not — see master spec §5.6): a --no-verify flag is forbidden."
```

### 3.6 New file `scripts/security_invariants.sh` (~80 lines)

The four-section CI script referenced from master spec §6.5, §7.2 and
multiple sub-specs.

```bash
#!/usr/bin/env bash
# Mechanical regression catches for the leakage taxonomy (threat model §7).
# Runs at pre-commit AND in CI. Fast: <1 s.
set -euo pipefail

REPO_ROOT="$(git rev-parse --show-toplevel 2>/dev/null || pwd)"
cd "$REPO_ROOT"

fail=0

# TM-LEAK.1 — tracing macros with secret-named fields.
# Catches: tracing::info!(?token, …) or tracing::warn!(api_key = %k.as_str(), …)
# The regex matches `?` or `%` followed by an identifier whose name
# matches the secret-name lexicon.
if grep -REn 'tracing::(info|warn|error|debug)!\s*\([^)]*[?%][a-zA-Z_][a-zA-Z0-9_]*\b' crates/ \
   | grep -E '\b(token|api_key|secret|password|hash|bearer|auth|url)\b' >/tmp/tm_leak1.txt 2>/dev/null; then
    echo "TM-LEAK.1 fail: tracing macro with a secret-named field." >&2
    cat /tmp/tm_leak1.txt >&2
    fail=1
fi

# Allowlist for L1: structured fields where the VALUE is non-secret.
# Example: `tracing::warn!(revoked_api_keys = count, …)` — count is fine.
# Such cases are listed by SHA in `scripts/tm_leak1.allowlist` (one line
# per `<file>:<line>` pair).
if [[ -f scripts/tm_leak1.allowlist ]] && [[ -s /tmp/tm_leak1.txt ]]; then
    if grep -vFf scripts/tm_leak1.allowlist /tmp/tm_leak1.txt > /tmp/tm_leak1_real.txt; then
        if [[ -s /tmp/tm_leak1_real.txt ]]; then
            fail=1
        else
            fail=0  # all hits were allowlisted
        fi
    fi
fi

# TM-LEAK.2 — derived-Debug heuristic. Look for `#[derive(...Debug...)]`
# followed within 30 lines by a String/Vec<u8>/PathBuf field whose name
# matches the secret-name lexicon. Multi-line grep via ripgrep-style
# context.
if grep -REn -A 30 '#\[derive\([^)]*\bDebug\b[^)]*\)\]' crates/ \
   | grep -E '\b(token|api_key|secret|password|hash|bearer|url)\s*:\s*(String|Vec<u8>|PathBuf|&str)' \
   > /tmp/tm_leak2.txt 2>/dev/null; then
    echo "TM-LEAK.2 warn: derived Debug on a struct with a secret-named String/PathBuf field." >&2
    cat /tmp/tm_leak2.txt >&2
    echo "If this is intentional (the Debug impl is manually redacted), add to scripts/tm_leak2.allowlist." >&2
    if [[ -f scripts/tm_leak2.allowlist ]]; then
        if grep -vFf scripts/tm_leak2.allowlist /tmp/tm_leak2.txt > /tmp/tm_leak2_real.txt; then
            if [[ -s /tmp/tm_leak2_real.txt ]]; then
                fail=1
            fi
        fi
    else
        fail=1
    fi
fi

# TM-SUPPLY.5 — assert web has no CDN script/link without integrity=.
if grep -REn '(cdnjs|jsdelivr|unpkg|cloudflare\.com/ajax)' web/ \
   --include='*.html' --include='*.tsx' --include='*.ts' \
   | grep -v 'integrity=' >/tmp/tm_supply5.txt 2>/dev/null; then
    if [[ -s /tmp/tm_supply5.txt ]]; then
        echo "TM-SUPPLY.5 fail: CDN asset reference without integrity= attribute." >&2
        cat /tmp/tm_supply5.txt >&2
        fail=1
    fi
fi

rm -f /tmp/tm_leak1.txt /tmp/tm_leak1_real.txt /tmp/tm_leak2.txt /tmp/tm_leak2_real.txt /tmp/tm_supply5.txt

exit $fail
```

Notes:

- The L1 / L2 regexes are intentionally **conservative** (over-trigger
  rather than under-trigger). False positives go in allowlist files
  (committed) keyed by file:line. The allowlist mechanism makes the
  guard sticky: a refactor that moves the line breaks the allowlist
  entry and forces a re-review.
- The TM-SUPPLY.5 check is currently a no-op (no CDN refs exist). Its
  job is to **stay** a no-op — i.e., catch any future contributor who
  adds a `<script src="https://cdn…">` without `integrity=`.

verify_perms() {
    # TM-SECRET.1: post-install sanity check. If a script edit ever
    # accidentally widens the chmod, the deploy aborts before the
    # service starts.
    local mode owner
    mode=$(stat -c %a "$ETC_DIR/gateway.toml")
    owner=$(stat -c %U:%G "$ETC_DIR/gateway.toml")
    if [[ "$mode" != "640" ]]; then
        echo "ERROR: $ETC_DIR/gateway.toml mode is $mode, expected 640" >&2
        exit 1
    fi
    if [[ "$owner" != "root:$GROUP" ]]; then
        echo "ERROR: $ETC_DIR/gateway.toml owner is $owner, expected root:$GROUP" >&2
        exit 1
    fi
}

```markdown
## Verifying git history is clean of secrets (one-time, before flipping the repo public)

`TM-SECRET.13` — pre-public scrub. The repo is private as of 2026-05-23;
before flipping it public, run:

    gitleaks detect --source . --log-level info --no-banner

For each hit:

- **False positive** (placeholder example, regression test fixture) —
  add to `.gitleaksignore` with the fingerprint format
  `<commit_sha>:<file_path>:<rule_id>` and a brief comment.
- **Real leak** — ROTATE the credential immediately via
  `revoked.toml` + SIGHUP (`TM-AUTH.4`). The leaked value is now dead;
  whether to scrub git history is a separate decision:
  - If the value is dead and the repo is small, leaving the historical
    blob is acceptable.
  - If the repo will be public and the value is recoverable from a third
    party (e.g., it was a CF API token someone could correlate), scrub
    with `git filter-repo --invert-paths --path <leaky-file>` or
    `git filter-repo --replace-text <patterns-file>`, then force-push
    once (announce in commit log).

The scrub is a manual ops step, not automated. Document the decision and
date in a one-line entry below this section.

## Installing the pre-commit hook (every contributor, one-time)

`TM-SECRET.8` — local hook. After cloning:

    bash scripts/install-pre-commit.sh

The script verifies that `gitleaks` and `cargo` are on PATH and installs
`.git/hooks/pre-commit`. Per master spec §5.6, **never** bypass with
`--no-verify`. If a hook produces a false positive, add an allowlist
entry (`scripts/tm_leak1.allowlist`, `scripts/tm_leak2.allowlist`, or
`.gitleaksignore`) — the allowlist itself is reviewed at PR time.
```

because `ApiKey::TryFrom<String>` validates the length (43 chars). The
placeholder string is intentionally exactly 43 chars in the right
alphabet so the parse succeeds and the test exercises the permissions
check, not the parser. The string is NOT a real key.

## 6. Risk-of-regression notes

### 6.1 Unix vs Windows asymmetry

The wrapper perms check is `#[cfg(unix)]`-gated. On Windows (the user's
dev box) the entire check is a no-op. Tests in
`config_perms_test.rs` use `#![cfg(unix)]` at the file level so they
compile to zero tests on Windows — no CI failure on `windows-latest`
if added later. Verified by the existing wrapper test pattern.

### 6.2 Existing wrapper tests that call `WrapperConfig::load`

On Unix, any existing test that writes a config to a temp dir and calls
`load` must now `set_permissions(0o600)` first. Grep result for
current callers:

- `crates/claude-phone-wrapper/tests/` — none currently load from disk;
  all construct `WrapperConfig` directly via struct literal. Safe.
- `crates/claude-phone-wrapper/src/main.rs` — production path; users
  already have a real config file with whatever perms they set. Tests
  do not exercise this path; integration test `e2e_test.rs` uses
  in-process state. Safe.

If a future test loads from disk, it must `set_permissions(0o600)`. The
forward-looking test message tells the developer exactly what to do.

### 6.3 gitleaks false positives in the existing tree

A pre-run on the current tree (verification step before commit 3) must
return clean — no current secret-shaped string in any committed file.
The known intentional placeholders are:

- `gateway-dev.toml` line 12 — `REPLACE_WITH_FRESHLY_GENERATED_BASE64URL_NO_PAD`.
  Already gitignored (`*-dev.toml` pattern in `.gitignore`).
- Test fixtures may embed similar 43-char strings — none currently
  detected. If added later, allowlist entry required.

### 6.4 CI cost increase

Adding `secrets-scan` job adds ~30-60 s to CI wall time (gitleaks scan
of the full history). Acceptable for a project this size; far below the
existing `cargo test --workspace` wall time. The job runs in parallel
with `rust`, `cargo-audit`, and `web` jobs.

### 6.5 Pre-commit hook performance

The four-step hook (fmt + clippy + gitleaks + invariants) on the current
tree runs in ~2-4 s on a cold cache, ~1 s on a warm cache. Per master
spec §6.5: ≤ 5 s total. Acceptable. If clippy ever blows the budget, the
fix is to scope clippy to staged files (`cargo clippy -p <crate>` based
on `git diff --staged --name-only`) rather than to relax the hook.

### 6.6 Forward-looking guards (do NOT remove without re-design)

- `// TM-SECRET.10:` comment in `wrapper/src/config.rs::load`.
- `// TM-SECRET.11:` comment in `gateway/src/error.rs` (both module doc
  and the 5xx arm).
- `// TM-SECRET.1:` annotations in `deploy/scripts/deploy.sh` at both
  chmod and `verify_perms`.
- `scripts/security_invariants.sh` — the regexes are conservative on
  purpose. If a refactor produces a false positive, add an allowlist
  entry rather than relaxing the regex.
- Tests #3, #4, #5, #6, #7 are invariant tests, not behaviour tests —
  failing means a guard was loosened, not a feature broke.

### 6.7 ApiKey deserialization interaction

`WrapperConfig` already deserializes `api_key` through `ApiKey::TryFrom<String>`
which validates the 43-char length and the base64url alphabet
(`TM-INPUT.7` GREEN, annotated by sub-spec 4.5). A malformed config
file thus fails at parse, before the perms check. That ordering is
intentional and correct: a 0600-permission attacker-controlled file is
still a fail; a 0644-permission valid-shape file fails the perms check
even though the parse would succeed. Both layers are independent.
