### 1.1 Out of scope here

- **`TM-RATE.5` 80% utilization alert** — that's a metrics-emit, owned by sub-spec **4.4** if/when it is added (master spec §3 currently has no 4.4 row for metrics; the alert is documented as `Accepted risk (deferred P2)` in the unit file comment). 4.9 sets the cap; observing the cap is a follow-up.
- **MAC (AppArmor / SELinux) profile** — `Accepted risk (deferred P2)` per threat model §9.18. Ubuntu's default AppArmor profile for systemd services covers the basics; a custom claude-phone profile adds 200 LOC of YAML for marginal gain given the systemd hardening already enforced.
- **Container image hardening (Distroless, CIS Docker benchmark)** — `deploy/Dockerfile.gateway` exists for compose-based deploys but the canonical path is bare-metal systemd. Docker-side hardening tracked separately if/when docker becomes the primary deploy target.
- **Off-host log shipping** — `Accepted risk (deferred P2)` per threat model §9.19. Single-host project; journald persistence + auditd local store is sufficient for the threat model.
- **HIDS (Wazuh, OSSEC)** — `Accepted risk (deferred P2)` per threat model §9.20. fail2ban + auditd give the active-defense layer needed at this asset value.

### 1.2 Open-question lock-ins specific to 4.9

- **LimitNOFILE value:** **8192**. Rationale: `max_sessions = 32` in `deploy/scripts/deploy.sh:56`; each session uses ≤ 4 FDs (wrapper WS + phone WS + pty master + epoll registration); peak FDs at full load ≈ 128. 8192 leaves >>50× headroom for transient handshake bursts and Caddy reverse-proxy FDs. Sane defaults; will be revisited if `max_sessions` is bumped.
- **MemoryMax:** **256M**. Steady-state memory of a Rust gateway with ~30 idle sessions is < 30 MB; 256M tolerates spike during a flash crowd or replay-buffer fill across many sessions. Tracked metric, can tighten if metrics show consistent < 50 MB.
- **CPUQuota:** **80%**. Leaves headroom on a single-core VPS for SSH and Caddy to coexist; on multi-core hosts this is per-effective-CPU which is fine.
- **fail2ban claude-phone jail bantime:** **24 h** after **5 failures in 10 min**. Same parameter shape as 4.6's `AuthRateLimiter` exponential backoff baseline, but operating on a separate timescale (process-restart-survivable, OS-level firewall ban vs in-process counter). Layered defense.
- **ufw allow-list source IPs for SSH:** operator-side; the apply script reads from `/etc/claude-phone/ssh-allowlist` (one IP/CIDR per line) and fails-loud if missing.
- **Cloudflare WAF rule definition:** operator-applied via dashboard; spec provides a checklist + Terraform-style reference snippet for documentation. No automated apply because CF API token scope for WAF rules is much broader than the read-only TLS settings token used in 4.3.

## 2. Detailed mitigation design

### 2.1 TM-INFRA.1 / .6 / .8 / .9 / .11 — systemd unit extension

**File:** `deploy/systemd/claude-phone-gateway.service` (modify in place)

Add to `[Service]` section, in same order as related directives for readability:

```ini
# --- TM-INFRA.1 — syscall filter (missing today) -------------------------
SystemCallFilter=@system-service
SystemCallFilter=~@privileged @resources
SystemCallErrorNumber=EPERM

# --- TM-INFRA.6 / TM-RATE.5 — resource caps ------------------------------
LimitNOFILE=8192
MemoryMax=256M
CPUQuota=80%

# --- TM-INFRA.8 — no core dumps ------------------------------------------
LimitCORE=0

# --- TM-INFRA.9 — confine writes -----------------------------------------
# ReadWritePaths is already set above; add an explicit ReadOnlyPaths
# for the install dir so a compromised gateway can't overwrite its own
# binary or web assets.
ReadOnlyPaths=/opt/claude-phone

# --- TM-INFRA.11 — user/group already set; reinforce via ReadOnlyPaths ---
# (covered by line above)
```

Each directive carries a `// TM-INFRA.<N>` comment (in INI style, `#`).

The `SystemCallFilter` policy uses `@system-service` (systemd's recommended default for daemons), then explicitly denies `@privileged` and `@resources` (which `@system-service` includes a permissive subset of). `SystemCallErrorNumber=EPERM` makes denied syscalls return permission-denied rather than terminating the process — important for debugging since a denied syscall typically signals a bug, not a real attack, and we want logs.

**Why not stricter `@network-io`-only?** Reviewed — `@network-io` excludes `clone`/`futex` and breaks tokio's spawn. The `@system-service` minus `@privileged @resources` recipe is the standard hardening recipe for Rust async services; tested in Debian/Ubuntu's own systemd profile recommendations.

### 2.2 TM-INFRA.2 — ufw apply script

**File:** `deploy/scripts/setup-ufw.sh` (new, executable)

```bash
#!/usr/bin/env bash
# Apply ufw rules for the claude-phone host. Idempotent — re-running just
# re-asserts the desired state.
#
# Pre-requisites:
#   1. /etc/claude-phone/ssh-allowlist exists, one IP/CIDR per line, no comments
#   2. ufw is installed (apt install ufw)
#   3. Cloudflare IP ranges fetched freshly:
#        curl -sSL https://www.cloudflare.com/ips-v4 > /etc/claude-phone/cf-ipv4
#        curl -sSL https://www.cloudflare.com/ips-v6 > /etc/claude-phone/cf-ipv6
#   4. Operator has reviewed the resulting `ufw status` output

set -euo pipefail

if [[ "$(id -u)" -ne 0 ]]; then
    echo "Run as root" >&2
    exit 1
fi

ALLOWLIST="/etc/claude-phone/ssh-allowlist"
CF4="/etc/claude-phone/cf-ipv4"
CF6="/etc/claude-phone/cf-ipv6"

for f in "$ALLOWLIST" "$CF4" "$CF6"; do
    if [[ ! -s "$f" ]]; then
        echo "Missing or empty: $f" >&2
        exit 1
    fi
done

# Default-deny.
ufw --force reset
ufw default deny incoming
ufw default allow outgoing

# SSH from allowlisted sources only.
while IFS= read -r src; do
    [[ -z "$src" || "$src" =~ ^# ]] && continue
    ufw allow proto tcp from "$src" to any port 22 comment "ssh allow $src"
done < "$ALLOWLIST"

# HTTPS from Cloudflare ranges only.
for src in $(cat "$CF4" "$CF6"); do
    ufw allow proto tcp from "$src" to any port 443 comment "cf $src"
done

ufw --force enable
ufw status verbose
```

The `comment` field is the audit anchor — `ufw status` output shows which rules came from which source, which is critical when the operator later debugs a "why am I locked out" scenario.

CF IP ranges are explicitly **fetched fresh by the operator** before each apply, not committed to repo. Rationale: CF IP ranges change quarterly-ish; committing them creates a maintenance burden and risks stale rules locking out CF traffic. The runbook in SECURITY.md tells the operator to refresh before re-running.

### 2.3 TM-INFRA.3 — sshd drop-in

**File:** `deploy/sshd/99-claude-phone.conf` (new, deployed to `/etc/ssh/sshd_config.d/`)

```
# TM-INFRA.3 — claude-phone sshd hardening. Drop-in form so distro-default
# /etc/ssh/sshd_config is untouched; 99-claude-phone.conf is loaded last.

PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
KbdInteractiveAuthentication no
ChallengeResponseAuthentication no

# Allow only operator user(s). Whitespace-separated list.

AllowUsers operator

MaxAuthTries 3
LoginGraceTime 30
ClientAliveInterval 300
ClientAliveCountMax 2

# Disable X11 / agent forwarding by default; reachable via -X if needed.
X11Forwarding no
AllowAgentForwarding no
AllowTcpForwarding no
GatewayPorts no

# Verbose logging for auditd / fail2ban consumption.
LogLevel VERBOSE
```

`deploy.sh` is extended (per §3 below) to install this drop-in and `sshd -t` validate before restarting sshd.

### 2.4 TM-INFRA.4 — fail2ban jails and filter

**File:** `deploy/fail2ban/jail.local` (new, deployed to `/etc/fail2ban/`)

```ini
# TM-INFRA.4 — claude-phone fail2ban jails.
# - sshd jail uses distro default filter.
# - recidive escalates repeat-offenders across all jails.
# - claude-phone jail watches the gateway's auth-failure log; filter
#   defined in filter.d/claude-phone.conf.

[DEFAULT]
banaction = ufw
backend = systemd

[sshd]
enabled = true
maxretry = 3
findtime = 10m
bantime = 1h

[recidive]
enabled = true
maxretry = 3
findtime = 1d
bantime = 1w

[claude-phone]
enabled = true
port = 443
filter = claude-phone
logpath = /var/log/syslog
maxretry = 5
findtime = 10m
bantime = 1d
```

**File:** `deploy/fail2ban/filter.d/claude-phone.conf` (new, deployed to `/etc/fail2ban/filter.d/`)

```ini
# TM-INFRA.4 — claude-phone filter. Parses the auth-failure JSON log from
# the gateway, format defined by sub-spec 4.2 §2.X. The gateway emits
# tracing JSON of the form:
#   { "level":"WARN", "fields":{ "event":"auth_failure", "ip":"<HOST>", ... } }
# We match on the event tag and capture the ip field.

[Definition]
failregex = ^.*"event":"auth_failure".*"ip":"<HOST>".*$
ignoreregex =

# Date format matches tracing-subscriber default RFC3339 emit.
datepattern = ^.*"timestamp":"%%Y-%%m-%%dT%%H:%%M:%%S
```

The filter is **defensive against log-format drift**: if 4.2's log shape changes, `fail2ban-regex /var/log/syslog deploy/fail2ban/filter.d/claude-phone.conf` will fail to match and the jail silently no-ops. A dedicated forward-looking test in 4.9 (§4.3) runs `fail2ban-regex` against a canned auth-failure log sample to assert the regex matches; CI fails if 4.2's log format changes without 4.9's filter being updated.

### 2.5 TM-INFRA.5 — auditd rules

**File:** `deploy/auditd/claude-phone.rules` (new, deployed to `/etc/audit/rules.d/`)

```
# TM-INFRA.5 — claude-phone auditd rules. Loaded after distro defaults.

# Watch the config dir — any write attempt is audited.
-w /etc/claude-phone/ -p wa -k claude-phone-config

# Watch the install dir — any write attempt to bin/ or web/ is audited.
-w /opt/claude-phone/ -p wa -k claude-phone-bin

# Watch the unit file — modifications by anything other than `deploy.sh`
# are suspicious.
-w /etc/systemd/system/claude-phone-gateway.service -p wa -k claude-phone-unit

# Watch the sshd drop-in.
-w /etc/ssh/sshd_config.d/99-claude-phone.conf -p wa -k claude-phone-sshd
```

`-p wa` audits writes and attribute changes. `-k claude-phone-*` keys make `ausearch -k claude-phone-config` queries straightforward for the operator.

### 2.6 TM-INFRA.7 — Cloudflare WAF rules

Cloudflare Free tier supports up to **5 custom WAF rules**. We use **3**, leaving 2 for the operator's future use.

Document the rules in `deploy/cloudflare/README.md` (append to existing file from 4.3):

```markdown
## Custom WAF rules (Free tier — 5 max)

Apply via Dashboard → Security → WAF → Custom rules.

| # | Name              | Expression                                                                                         | Action |
|---|-------------------|----------------------------------------------------------------------------------------------------|--------|
| 1 | block-scan-paths  | `(http.request.uri.path contains "/.git" or http.request.uri.path contains "/.env" or http.request.uri.path contains "/wp-admin" or http.request.uri.path contains "/vendor/phpunit" or http.request.uri.path contains "/xmlrpc.php")` | Block  |
| 2 | block-unknown-api | `http.request.uri.path matches "^/api/(?!wrapper$|phone/)"`                                       | Block  |
| 3 | rate-wrapper      | `http.request.uri.path eq "/api/wrapper"` — Rate-limit 10/min/IP                                  | Block (1 h) |

Rule #2 enforces our endpoint allow-list at the CDN layer — a defense-in-depth
mirror of the origin's route mux. Any new `/api/*` route added in the
gateway requires this rule to be updated, which is a deliberate friction
point: new endpoints get reviewed.

Rule #3 is the CDN-layer mirror of `TM-RATE.2` and `TM-RATE.1`; even if a
flood reaches CF, it gets stopped before hitting our origin.
```

### 2.8 TM-INFRA.10 — loopback-only binding verification

Extend `deploy/scripts/post_deploy_verify.sh` (introduced by 4.3) with one block:

```bash
# --- TM-INFRA.10 — gateway binds only to 127.0.0.1 ----------------------
echo "[4/N] gateway binding ..."
binding=$(ss -tlnp 2>/dev/null \
    | grep claude-phone-gateway \
    | awk '{print $4}')
if [[ "$binding" =~ ^127\.0\.0\.1:|^\[::1\]: ]]; then
    green "Gateway bound to loopback only ($binding)"
else
    red   "Gateway bound to non-loopback: $binding"
    exit 1
fi
```

The check is run by the operator after each deploy; if the binding ever changes (e.g., a config typo sets `bind_addr = "0.0.0.0:8080"`), the verifier fails loudly.

### 2.9 `deploy/scripts/deploy.sh` extension

Append after `install_systemd`:

```bash
install_sshd_dropin() {
    if [[ -f "$REPO_DIR/deploy/sshd/99-claude-phone.conf" ]]; then
        install -m 0644 "$REPO_DIR/deploy/sshd/99-claude-phone.conf" \
            /etc/ssh/sshd_config.d/

}

install_fail2ban() {
    if [[ -f "$REPO_DIR/deploy/fail2ban/jail.local" ]]; then
        install -m 0644 "$REPO_DIR/deploy/fail2ban/jail.local" /etc/fail2ban/
        install -m 0644 "$REPO_DIR/deploy/fail2ban/filter.d/claude-phone.conf" \
            /etc/fail2ban/filter.d/
        systemctl reload fail2ban
    fi
}

install_auditd() {
    if [[ -f "$REPO_DIR/deploy/auditd/claude-phone.rules" ]]; then
        install -m 0640 "$REPO_DIR/deploy/auditd/claude-phone.rules" \
            /etc/audit/rules.d/
        augenrules --load
    fi
}

install_journald() {
    if [[ -f "$REPO_DIR/deploy/journald/99-claude-phone.conf" ]]; then
        install -d /etc/systemd/journald.conf.d
        install -m 0644 "$REPO_DIR/deploy/journald/99-claude-phone.conf" \
            /etc/systemd/journald.conf.d/
        systemctl restart systemd-journald
    fi
}
```

### 3.3 Operator action items (added to `SECURITY.md`)

After deploy:

1. Populate `/etc/claude-phone/ssh-allowlist`.
2. Run `curl -sSL https://www.cloudflare.com/ips-v4 > /etc/claude-phone/cf-ipv4` and v6 equivalent.
3. Edit `99-claude-phone.conf` `AllowUsers` to match operator's account.
4. Run `deploy/scripts/setup-ufw.sh`.
5. Apply Cloudflare WAF rules via dashboard (§2.6).
6. Run `deploy/scripts/post_deploy_verify.sh`.

These are deliberately **NOT automated** because they encode operator-specific values (IPs, usernames) that must not be committed. The runbook is the contract.

### 3.4 Rollback story

Each new file is self-contained. To revert a single mitigation:

- systemd directives — comment out the line + `systemctl daemon-reload && systemctl restart claude-phone-gateway`.
- ufw — `ufw --force reset && ufw disable`.
- sshd drop-in — `rm /etc/ssh/sshd_config.d/99-claude-phone.conf && systemctl reload sshd`.
- fail2ban — `systemctl stop fail2ban && rm /etc/fail2ban/jail.local`.
- auditd — `rm /etc/audit/rules.d/claude-phone.rules && augenrules --load`.
- journald — `rm /etc/systemd/journald.conf.d/99-claude-phone.conf && systemctl restart systemd-journald`.
- CF WAF — disable rule in dashboard.

Rollback is **per-mitigation**, never repo-wide — the rollback.sh script from earlier deploys is for the gateway binary itself, not for these OS-level changes.

### 4.3 Forward-looking fail2ban regression test

**File:** `scripts/fail2ban_filter_test.sh` (new, executable)

```bash
#!/usr/bin/env bash
# TM-INFRA.4 — assert the claude-phone filter matches the auth-failure log
# format defined by sub-spec 4.2. If 4.2's format ever changes without
# updating the filter, this test fails in CI before deploy.

set -euo pipefail

if ! command -v fail2ban-regex >/dev/null; then
    echo "fail2ban-regex not installed; install with: apt install fail2ban"
    exit 2
fi

SAMPLE=$(mktemp)
cat > "$SAMPLE" <<'EOF'
2026-05-23T12:34:56.000Z {"timestamp":"2026-05-23T12:34:56.000Z","level":"WARN","fields":{"event":"auth_failure","ip":"203.0.113.42","correlation_id":"01HX...","reason":"invalid_api_key"}}
2026-05-23T12:35:00.000Z {"timestamp":"2026-05-23T12:35:00.000Z","level":"INFO","fields":{"event":"auth_success","ip":"203.0.113.99","correlation_id":"01HY..."}}
EOF

# Expect 1 match (the auth_failure line) and 0 matches on the auth_success.
result=$(fail2ban-regex --print-no-missed "$SAMPLE" \
    deploy/fail2ban/filter.d/claude-phone.conf 2>&1 || true)

if ! echo "$result" | grep -qE "Failregex.*1 match"; then
    echo "Expected 1 match in sample; got:"
    echo "$result"
    rm -f "$SAMPLE"
    exit 1
fi

if echo "$result" | grep -qE "203\.0\.113\.99"; then
    echo "ERROR: filter matched auth_success line (should only match auth_failure)"
    rm -f "$SAMPLE"
    exit 1
fi

rm -f "$SAMPLE"
echo "fail2ban filter regression: OK"
```

Hook into `scripts/security_invariants.sh`:

```bash
echo "[security_invariants] fail2ban filter regression ..."
./scripts/fail2ban_filter_test.sh || {
    # Soft-fail in environments without fail2ban (e.g., dev workstations).
    # CI runner has fail2ban available; this is a real assert there.
    echo "(fail2ban-regex unavailable — skipping)"
}
```

## 6. Risk-of-regression analysis

| Change                                            | Risk                                                                                                                  | Guard                                                                                                                                |
|---------------------------------------------------|-----------------------------------------------------------------------------------------------------------------------|--------------------------------------------------------------------------------------------------------------------------------------|
| `SystemCallFilter=@system-service`                | A tokio internal call (e.g., `prctl`, `clone3`) might be denied, causing gateway to crash on syscall.                 | `@system-service ~@privileged ~@resources` is the standard recipe for Rust async services; tested via `cargo test --workspace` after applying. If a denied syscall appears, gateway logs `Operation not permitted` and operator widens the filter; `SystemCallErrorNumber=EPERM` ensures soft-fail not SIGSYS. |
| `LimitNOFILE=8192`                                | Too low → connection storms drop legitimate users.                                                                    | Sized 50× above peak (§1.2). TM-RATE.5 observability deferred but `journalctl --grep "Too many open files"` is the operator's canary. |
| `MemoryMax=256M`                                  | Memory leak in a future code change OOMs the gateway under cap.                                                       | OOM is loud (systemd logs, gateway restarts); preferred to silent unbounded growth on a $20/mo VPS. Tune up if observed.             |
| ufw default-deny                                  | Operator locks themselves out by running setup-ufw.sh without populating ssh-allowlist.                               | Script fails fast on missing/empty allowlist (§2.2). Documented in SECURITY.md runbook: "test in screen/tmux, have a console fallback." |
| sshd `AllowUsers operator`                        | Operator runs `setup-sshd` without editing the `operator` placeholder; SSH locks out.                                 | Drop-in file is human-edited by operator before deploy; documented step. Alternative considered (auto-detect current user) rejected — too magic. |
| fail2ban `claude-phone` jail filter regex          | 4.2 log format change silently breaks the jail.                                                                       | `scripts/fail2ban_filter_test.sh` runs in CI, fails red if filter no longer matches the canned sample. 4.2 sub-spec already calls out this dependency. |
| auditd writes to syslog                           | Disk fill on a low-disk host.                                                                                          | `SystemMaxUse=512M` (journald) and auditd default disk_full_action; documented in SECURITY.md.                                       |
| CF WAF rule #2 (`unknown-api`)                    | New `/api/*` route silently blocked when added.                                                                       | Intentional friction. Documented in §2.6. The blocker is the *point*.                                                                |
| Persistent journal                                | Disk fill.                                                                                                              | `SystemMaxUse=512M` cap.                                                                                                              |
