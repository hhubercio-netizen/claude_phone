### 1.2 TM-WS.9 cross-spec note

Master §3 is a *concern matrix*, not an ownership ledger.

### 1.3 Open-question lock-ins specific to 4.3

- **CT monitoring frequency:** daily at 06:00 UTC. Rationale: certificates are issued in well-known time windows (Let's Encrypt renews around 60-day mark, Cloudflare on edge-cert rollovers), and one missed day is recoverable; daily is also the lightest cron that won't suspend (GH Actions suspends cron on inactive repos at >60 days, which won't happen during active hardening work).
- **testssl baseline:** **no HIGH or CRITICAL findings** (the `--severity HIGH` flag); MEDIUM is allowed because Caddy + CF combinations sometimes emit MEDIUM advisories on legacy options that are not actually negotiable end-to-end. The verifier emits MEDIUM findings as a log line, not a failure.
- **CF TLS mode source of truth:** Cloudflare API (`/zones/{id}/settings/ssl`). Verified at deploy time, not as a CI job, because the API token is sensitive and shouldn't live in repo CI.

## 2. Detailed mitigation design

### 2.1 TM-TLS.1 — TLS 1.3 only at Caddy

Caddyfile today (`deploy/caddy/Caddyfile:8-9`):

```
claude-phone.pl:443 {
    tls /etc/caddy/cert.pem /etc/caddy/key.pem
```

Caddy's default `tls` directive accepts TLS 1.2 and 1.3. We want **TLS 1.3 only**: zero-RTT path with forward secrecy by default, no CBC-mode legacy, no RSA key exchange. Modern browsers (Chrome ≥ 70, Firefox ≥ 63, Safari ≥ 12.1, Edge ≥ 79) all do 1.3; the install base our app cares about (phones with modern browsers) is 100% covered.

```
claude-phone.pl:443 {
    tls /etc/caddy/cert.pem /etc/caddy/key.pem {
        protocols tls1.3
    }
```

The `protocols` sub-directive locks the version floor and ceiling to 1.3.

### 2.2 TM-TLS.2 — HSTS parity between Caddy and gateway

Caddyfile today (`deploy/caddy/Caddyfile:36`):

```
Strict-Transport-Security "max-age=31536000; includeSubDomains"
```

Gateway (`crates/claude-phone-gateway/src/http.rs:113-115`):

```rust
HeaderValue::from_static("max-age=63072000; includeSubDomains; preload"),
```

Caddy advertises 1 year and no preload; gateway advertises 2 years with preload. Browsers cache the *highest* policy they've seen — so a user who first lands while Caddy headers shadow gateway headers (e.g., Caddy adds the header after gateway, or there's intermediate proxying that strips gateway header) gets the weaker policy until next visit.

**Change Caddyfile to:**

```
Strict-Transport-Security "max-age=63072000; includeSubDomains; preload"
```

Exact-match with gateway's value. Now every layer says the same thing.

`preload` in the value is the prerequisite for submitting to hstspreload.org (TM-TLS.3, deferred). Without `preload` in the header, the submission is rejected; with it, the policy is honored only after a 30-day soak.

### 2.3 TM-TLS.4 — CT monitoring via crt.sh

**File:** `.github/workflows/ct-monitor.yml` (new)

```yaml
name: CT monitor
on:
  schedule:
    - cron: '0 6 * * *'   # 06:00 UTC daily
  workflow_dispatch:

jobs:
  monitor:
    runs-on: ubuntu-latest
    permissions:
      contents: read
    steps:
      - uses: actions/checkout@v4

      - name: Restore baseline
        id: cache-baseline
        uses: actions/cache@v4
        with:
          path: .ct-baseline
          key: ct-baseline-claude-phone-pl
          restore-keys: |
            ct-baseline-claude-phone-pl

      - name: Run CT monitor
        run: ./scripts/ct_monitor.sh

      - name: Save updated baseline
        if: success()
        uses: actions/cache/save@v4
        with:
          path: .ct-baseline
          key: ct-baseline-claude-phone-pl-${{ github.run_id }}
```

**File:** `scripts/ct_monitor.sh` (new, executable)

```bash
#!/usr/bin/env bash
# TM-TLS.4 — daily check of CT logs for claude-phone.pl. Fails the workflow
# if a certificate appears in CT that is not in our local baseline. Operator
# investigates by visiting https://crt.sh/?q=claude-phone.pl and confirming
# whether the new cert was a renewal (expected) or a misissuance (incident).

set -euo pipefail

DOMAIN="${CT_DOMAIN:-claude-phone.pl}"
BASELINE="${CT_BASELINE:-.ct-baseline}"
API="https://crt.sh/?q=${DOMAIN}&output=json"

# crt.sh occasionally 503s or returns truncated JSON. Retry up to 3 times
# with backoff so a transient failure doesn't page the operator.
for attempt in 1 2 3; do
    if response=$(curl -fsSL --max-time 30 "$API"); then
        break
    fi
    if [ "$attempt" -eq 3 ]; then
        echo "::error::crt.sh API failed after 3 attempts"
        exit 2
    fi
    sleep $((attempt * 5))
done

today=$(echo "$response" | jq -r '.[].id' | sort -u)

if [ ! -s "$BASELINE" ]; then
    echo "$today" > "$BASELINE"
    count=$(echo "$today" | wc -l)
    echo "Created baseline with $count CT entries"
    exit 0
fi

new=$(comm -23 <(echo "$today") <(sort -u "$BASELINE"))
if [ -n "$new" ]; then
    echo "::error::New CT entries for $DOMAIN — investigate:"
    echo "$new"
    echo "::error::View: https://crt.sh/?q=$DOMAIN"
    exit 1
fi

echo "No new CT entries for $DOMAIN ($(echo "$today" | wc -l) total)"
# Refresh baseline with today's set so removed (expired) entries don't stick.
echo "$today" > "$BASELINE"
```

Operator runbook (one paragraph, added to `SECURITY.md` from 4.7): when the workflow fails, visit https://crt.sh/?q=claude-phone.pl and check that the new entry is a Let's Encrypt or Cloudflare renewal you expected. If it is **not** — for example, a "DigiCert" or "Sectigo" issuer when you only use LE — treat as `Asset #5: Domain reputation` incident per `INCIDENT_RESPONSE.md`.

Failure mode handled by §1.3 (3-retry on crt.sh transient errors).

### 2.4 TM-TLS.6 / .7 / .8 — post-deploy verifier

**File:** `deploy/scripts/post_deploy_verify.sh` (new, executable)

Runs after `deploy.sh` finishes pushing the artifact. Three checks:

```bash
#!/usr/bin/env bash
# Run after deploy.sh. Asserts TLS-layer mitigations are actually in effect
# on the live endpoint. Exit 0 on full success; exit 1 on any HIGH/CRITICAL
# finding; exit 2 on infrastructure error (missing tools, unreachable host).
#
# Idempotent — safe to re-run from the operator shell.

set -euo pipefail

HOST="${VERIFY_HOST:-claude-phone.pl}"
PORT="${VERIFY_PORT:-443}"
CF_ZONE_ID="${CLOUDFLARE_ZONE_ID:-}"
CF_API_TOKEN="${CLOUDFLARE_API_TOKEN:-}"

red()   { printf '\033[31m%s\033[0m\n' "$*"; }
green() { printf '\033[32m%s\033[0m\n' "$*"; }
warn()  { printf '\033[33m%s\033[0m\n' "$*"; }

# --- TM-TLS.6 OCSP stapling -----------------------------------------------
echo "[1/3] OCSP stapling on $HOST:$PORT ..."
staple=$(
    echo | openssl s_client -connect "$HOST:$PORT" -servername "$HOST" \
        -status -tls1_3 2>/dev/null \
        | awk '/OCSP response:/{flag=1; next} /---/{flag=0} flag'
)
if echo "$staple" | grep -q "Cert Status: good"; then
    green "OCSP staple present and good"
else
    red   "OCSP staple missing or not good"
    exit 1
fi

# --- TM-TLS.7 testssl HIGH/CRITICAL ---------------------------------------
echo "[2/3] testssl scan on $HOST ..."
# testssl docker image pinned by digest. Update with:
#   docker pull drwetter/testssl.sh:3.2 && docker inspect ... | jq -r .RepoDigests
TESTSSL_IMAGE="drwetter/testssl.sh@sha256:<PIN_AT_FIRST_RUN>"
if ! docker run --rm "$TESTSSL_IMAGE" --color 0 --severity HIGH \
        --quiet "$HOST" > /tmp/testssl.out 2>&1; then
    red "testssl: HIGH or CRITICAL finding"
    cat /tmp/testssl.out
    exit 1
fi
green "testssl: no HIGH/CRITICAL"
# Surface MEDIUM as warning, not failure (§1.3).
if grep -E "MEDIUM" /tmp/testssl.out >/dev/null 2>&1; then
    warn "testssl: MEDIUM findings — review"
    grep "MEDIUM" /tmp/testssl.out
fi

# --- TM-TLS.8 Cloudflare TLS mode = strict --------------------------------
echo "[3/3] Cloudflare TLS mode for zone $CF_ZONE_ID ..."
if [ -z "$CF_ZONE_ID" ] || [ -z "$CF_API_TOKEN" ]; then
    warn "CLOUDFLARE_ZONE_ID or CLOUDFLARE_API_TOKEN unset; skipping CF check"
else
    mode=$(curl -fsSL \
        "https://api.cloudflare.com/client/v4/zones/$CF_ZONE_ID/settings/ssl" \
        -H "Authorization: Bearer $CF_API_TOKEN" \
        | jq -r '.result.value')
    if [ "$mode" = "strict" ]; then
        green "Cloudflare TLS mode: strict"
    else
        red   "Cloudflare TLS mode: $mode (expected: strict)"
        exit 1
    fi
fi

green "Post-deploy TLS verification: OK"
```

**Pinning the testssl docker image:** `<PIN_AT_FIRST_RUN>` is the only intentional placeholder in this sub-spec, because a docker digest is environment-specific and is filled in by the operator the first time they run the script. The script will error loudly if left unfilled (docker pull will fail on `<PIN_AT_FIRST_RUN>`), and the inline comment documents how to compute the value. Recorded as a follow-up acceptable per master spec §10 quality bar.

**Wire into `deploy/scripts/deploy.sh`:** at the end of the script, after `verify_perms`, add:

```bash
echo "Running post-deploy TLS verification ..."
./deploy/scripts/post_deploy_verify.sh
```

### 3.1 No Rust code changes

4.3 is deliberately code-light. All mitigations live in deploy/ops territory (Caddy, CF, CT, testssl). The single Rust-side concern — `TM-WS.9` production-config validation — is owned by 4.13 (§1.2 above), not by 4.3.

This keeps 4.3's blast radius confined to deploy/ and .github/, where the existing test harness is operator-run rather than `cargo test`-run. The post-deploy verifier is the test.

### 3.2 No new Rust/npm deps

`jq`, `curl`, `openssl s_client`, `docker` are operator-side tools that must exist on the deploy host. The CI runner uses Ubuntu-latest which has all four pre-installed.

### 3.3 testssl image pin lifecycle

The pin in `post_deploy_verify.sh` is filled in **on first deploy run** by the operator. Pin update cadence: aligned with `cargo audit` — when 4.7's CI audit triggers, the operator also runs:

```
docker pull drwetter/testssl.sh:3.2
docker inspect drwetter/testssl.sh:3.2 | jq -r '.[0].RepoDigests[0]'
```

…and updates the `TESTSSL_IMAGE=...@sha256:...` line, committing the change separately (per master spec rule: cosmetic / version bumps separate from security changes).

### 3.4 Idempotence and re-run safety

- `ct_monitor.sh` re-running on the same day produces empty diff → exit 0. Re-runnable any time.
- `post_deploy_verify.sh` is read-only against the live infrastructure. Re-runnable any time without side effects.
- The CT baseline is cumulative (only adds; refresh prunes expired). Manual reset is a one-line `rm .ct-baseline`.

### 3.5 Failure-mode taxonomy

| Failure                              | Script exit code | Meaning                                                                       |
|--------------------------------------|------------------|-------------------------------------------------------------------------------|
| crt.sh 503 / timeout (3 retries)     | 2                | Infrastructure flake — re-run cron next day; do not page.                     |
| New CT entry vs baseline             | 1                | Real or expected event — operator investigates.                                |
| OCSP staple absent                   | 1                | Caddy regression or CA outage — investigate immediately.                      |
| testssl HIGH/CRITICAL                | 1                | Real security finding — rollback if recent change; page operator.             |
| CF TLS mode != strict                | 1                | Configuration drift — re-set in dashboard, re-verify.                          |
| Missing CF env vars                  | 0 (with warn)    | Skipped — not a failure, just a soft check.                                    |
