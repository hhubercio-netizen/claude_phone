## 2. Threat-by-threat narrative

### 2.1 TM-TEST.1 — cargo-fuzz

A new workspace member `crates/claude-phone-fuzz` (path-only, not published) with
three fuzz targets:

```text
crates/claude-phone-fuzz/
├── Cargo.toml            # [package] cargo-fuzz = true; [package.metadata] cargo-fuzz = true
├── fuzz_targets/
│   ├── control_message.rs   # arbitrary &[u8] → serde_json::from_slice::<ControlMessage>
│   ├── session_token.rs     # arbitrary &str  → SessionToken::from_str
│   └── api_key.rs           # arbitrary &str  → ApiKey::try_from
```

Each target shape:

```rust
// crates/claude-phone-fuzz/fuzz_targets/control_message.rs
#![no_main]
use libfuzzer_sys::fuzz_target;
use claude_phone_shared::protocol::ControlMessage;

fuzz_target!(|data: &[u8]| {
    // We don't care about the result; only that nothing panics, allocates
    // unboundedly, or infinite-loops on malformed input.
    let _ = serde_json::from_slice::<ControlMessage>(data);
});
```

**CI integration.** A separate GitHub Actions job `fuzz-smoke` runs each target
for 30 seconds (`cargo fuzz run <target> -- -max_total_time=30`) on every push
to `main` only (not on PRs, to keep PR CI fast). Failures upload the offending
corpus as a workflow artifact.

```yaml
  fuzz-smoke:
    name: cargo fuzz (smoke)
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@nightly
      - uses: Swatinem/rust-cache@v2
      - run: cargo install --locked cargo-fuzz
      - name: Fuzz control_message
        run: cargo fuzz run control_message -- -max_total_time=30
        working-directory: crates/claude-phone-fuzz
      - name: Fuzz session_token
        run: cargo fuzz run session_token -- -max_total_time=30
        working-directory: crates/claude-phone-fuzz
      - name: Fuzz api_key
        run: cargo fuzz run api_key -- -max_total_time=30
        working-directory: crates/claude-phone-fuzz
      - name: Upload corpus on failure
        if: failure()
        uses: actions/upload-artifact@v4
        with:
          name: fuzz-corpus-${{ github.run_id }}
          path: crates/claude-phone-fuzz/fuzz/artifacts/
```

`cargo fuzz` requires nightly Rust; this is the **only** job in the workflow
that uses nightly. The rest of CI stays on stable.

The 30 s budget is intentionally small — the value is "any new crash reproduces
on first PR" not "deep coverage". For deep coverage the operator runs
`cargo fuzz run <target>` locally with a larger budget.

### 2.4 TM-TEST.4 — pentest-style e2e

New file `crates/claude-phone-gateway/tests/pentest_e2e.rs`. Six pentest
scenarios:

| Scenario                                    | Setup                                                                                                                       | Assertion                                                                                          |
|---------------------------------------------|-----------------------------------------------------------------------------------------------------------------------------|----------------------------------------------------------------------------------------------------|
| Malformed JSON ControlMessage on wrapper WS | Spawn gateway, open WS to `/api/wrapper`, send `Text("{")` before hello                                                      | Connection closed with `ProtocolViolation`; no panic in gateway logs                                |
| Oversized hello on wrapper WS                | Send a `WrapperHello` text frame > 64 KiB                                                                                    | Connection closed; gateway still responsive to other clients                                       |
| Replay session token attack                  | Wrapper A registers token T, disconnects, attacker connects with same T as wrapper                                            | Second registration succeeds (re-registration is allowed) but receives a fresh session — no state from A's session leaks |
| Cross-session leak attempt                   | Wrapper A on token T, Phone B on token T2 (different); B sends frames addressed to T's session                                | B's frames are dropped (not routed to A's session); B receives `SessionNotFound` or similar         |
| Origin-spoof attack                          | Production config (`public_origin = Some("https://prod.example")`); phone WS upgrade with `Origin: https://evil.example`     | 403 close — never reaches application code                                                          |
| Concurrent same-token registration race       | Two `register_wrapper` calls with the *same* token in flight                                                                  | One succeeds, the other gets `SessionTaken`. Registry is consistent post-test.                       |

Each scenario gets a same-line `// TM-TEST.4` reference comment. The file
uses a shared `spawn_test_gateway` helper (modeled on the existing
`crates/claude-phone-gateway/tests/leakage_test.rs:15` helper) so test
isolation works.

### 2.5 TM-TEST.5 — forward-looking grep gates

Two scripts, both wired into CI in commit 3:

**`scripts/check_tracing_secrets.sh`:**

```bash
#!/usr/bin/env bash
# TM-LEAK.1 / TM-TEST.5
# Fail CI if any tracing macro logs a secret-named field that is not wrapped
# in our redacting types. The heuristic is: tracing::{info,warn,error,debug,trace}!
# call sites whose argument list mentions `api_key`, `token`, `session_token`,
# or `password` and does NOT also reference `redact`, `ApiKey(`, or
# `SessionToken(`.
set -euo pipefail

# Cross-platform note: -P (PCRE) is rg's default; the lookbehind below is
# emulated with grep -v rather than a true lookbehind to stay portable.
bad=$(rg --no-heading -nU \
    'tracing::(info|warn|error|debug|trace)!\([^)]*\b(api_key|token|session_token|password)\b[^)]*\)' \
    crates/*/src \
    | grep -v -E 'redact|ApiKey\(|SessionToken\(' \
    || true)

if [ -n "$bad" ]; then
    echo "Found tracing! call sites that may leak secrets (TM-LEAK.1):"
    echo "$bad"
    echo ""
    echo "To silence a false positive, refactor the call to use the redacting"
    echo "types or add a same-line // TM-LEAK.1: safe — <reason> comment."
    exit 1
fi
```

**`scripts/check_debug_derive_secrets.sh`:**

```bash
#!/usr/bin/env bash
# TM-LEAK.2 / TM-TEST.5
# Heuristic gate: a struct that has a field named `api_key`, `token`,
# `session_token`, `password` AND `#[derive(...Debug...)]` must NOT use
# the auto-derive — it must hand-implement `fmt::Debug` to redact. This
# catches the easy "add a field then forget" mistake.
#
# Skips files that contain "// TM-LEAK.2: hand-written Debug" on the
# preceding line of the derive.
set -euo pipefail

bad=$(rg --no-heading -nU --multiline \
    '#\[derive\([^)]*\bDebug\b[^)]*\)\]\s*\n\s*(pub\s+)?struct\s+\w+\s*\{[^}]*\b(api_key|token|session_token|password)\b' \
    crates/*/src \
    || true)

if [ -n "$bad" ]; then
    echo "Found auto-derived Debug on a struct that contains a secret-named field (TM-LEAK.2):"
    echo "$bad"
    echo ""
    echo "Implement Debug by hand and redact the secret field, or wrap the"
    echo "field in a type whose Debug is already redacting (ApiKey, SessionToken)."
    exit 1
fi
```

Both scripts run in the existing `rust` CI job:

```yaml
      - run: bash scripts/check_tracing_secrets.sh
      - run: bash scripts/check_debug_derive_secrets.sh
```

These are **heuristic** gates. They are not a replacement for the typed
redaction in 4.7 (`ApiKey`, `SessionToken`); they catch the regression mode
where someone bypasses the typed wrappers.

### 2.6 TM-TEST.6 — chaos test

`crates/claude-phone-gateway/tests/chaos_test.rs`. Three subtests:

1. **Wrapper task panics mid-session.** Spawn a session with wrapper and phone
   both attached. Force the wrapper task to terminate (via cancel-token or by
   dropping the wrapper-side WS). Assert: phone receives a `PeerDown` /
   `PeerStatus { peer_connected: false }` notification within 5 s, session
   stays alive (idle timeout takes over).
2. **Phone task panics mid-session.** Symmetric. Wrapper receives `PeerDown`.
3. **Many sessions; random subset killed.** Spawn `N=20` sessions, randomly
   kill `k=10` of them via cancel. Wait for sweeper. Assert:
   `registry.len() == 10` and no residual mpsc senders linger.

Each subtest annotated `// TM-TEST.6`.

The chaos test runs under `tokio::test(flavor = "multi_thread", worker_threads = 4)`.
It is slow (~5–10 s per subtest); it lives in the `tests/` directory but is
gated to the standard `cargo test` run — no separate CI job.

### 2.10 TM-LEAK.4 — coverage matrix

`scripts/tm_coverage.sh` is the centerpiece of this sub-spec. It walks
`docs/superpowers/specs/2026-05-23-threat-model.md` and extracts every
`TM-CAT.N` identifier, then greps the entire workspace (code + tests +
scripts + sub-specs) for that identifier. Each TM-CAT.N must appear in at
least one of:

- A code or test comment (`// TM-RATE.1`, `# TM-RATE.1`, etc.)
- A test name that includes the identifier (`tm_rate_1_*`)
- A script comment
- A sub-spec section heading or explicit `mitigates: TM-CAT.N` reference

If any identifier has zero matches outside of the threat-model and the
master design, CI fails with the list of uncovered IDs.

```bash
#!/usr/bin/env bash
# scripts/tm_coverage.sh
# TM-LEAK.4 — Every TM-CAT.N from the threat model must be referenced
# somewhere outside the threat model itself (code comment, test name,
# script, sub-spec). Catches "fix the bug, remove the test/comment,
# nobody notices the mitigation is gone".
set -euo pipefail

TM_FILE="docs/superpowers/specs/2026-05-23-threat-model.md"
DESIGN_FILE="docs/superpowers/specs/2026-05-23-security-hardening-design.md"

# Extract every TM-<CATEGORY>.<NUMBER> identifier from the threat model.
ids=$(rg -o 'TM-[A-Z]+\.[0-9]+' "$TM_FILE" | sort -u)

missing=()
for id in $ids; do
    # Search everywhere EXCEPT the threat model and the master design
    # (those are where the IDs are defined; finding them there proves nothing).
    hits=$(rg -l "$id" \
        --glob '!docs/superpowers/specs/2026-05-23-threat-model.md' \
        --glob '!docs/superpowers/specs/2026-05-23-security-hardening-design.md' \
        || true)
    if [ -z "$hits" ]; then
        missing+=("$id")
    fi
done

if [ ${#missing[@]} -gt 0 ]; then
    echo "The following threat-model IDs have no implementation or test reference:"
    for id in "${missing[@]}"; do
        echo "  - $id"
    done
    echo ""
    echo "Either implement the mitigation, add a test that references the ID,"
    echo "or update the threat model to mark the ID as DEFERRED / WONTFIX."
    exit 1
fi

echo "All $(echo "$ids" | wc -l | tr -d ' ') threat-model IDs have at least one workspace reference."
```

This gate is what enforces the convention seen in 4.6 / 4.13 / 4.14 specs
where every mitigation requires a `// TM-CAT.N` reference comment in the
code it modifies. Without this gate, a future refactor could remove the
mitigation and the only artifact left would be the threat-model entry —
which is exactly the failure mode this gate prevents.

CI step in commit 2:

```yaml
      - run: bash scripts/tm_coverage.sh
```

## 8. Failure modes

| Failure                                                                                                          | Symptom                                                                                                  | Mitigation                                                                                                                                  |

| `tm_coverage.sh` flags a DEFERRED-MINIMAL ID (e.g., TM-SUPPLY.3, TM-INFRA.10 if deferred)                          | CI red for an ID we never intended to implement                                                          | Update threat-model.md row to status `WONTFIX` or `DEFERRED` and ensure the script accepts those by recognising the explicit status row in the threat model. Implementation detail for the plan agent. |
| `check_tracing_secrets.sh` false-positives on a safe call site                                                    | CI red on a legitimate change                                                                            | Use the documented escape hatch — `// TM-LEAK.1: safe — <reason>` comment, or refactor to the redacting type.                                |

Do NOT silence the proptest. (If the panic is in a third-party crate, file upstream.)         |

## 9. Backwards compatibility and migration

- **No production code changes.** This sub-spec is tests + CI only.
- **CI runtime growth.** proptest: ~1–3 s. pentest_e2e: ~5–10 s. chaos: ~10–15 s. fuzz-smoke: 30 s × 3 = 90 s (main-branch only, parallel job). Heuristic grep gates: <1 s each. tm_coverage: ~2 s.
  Total PR CI growth: ~20–30 s. Acceptable.
- **Developer workstation impact.** `cargo test --workspace` now runs chaos
  + pentest + proptest in addition to existing tests. Total local test time
  grows by ~30 s. Fuzz is opt-in (requires nightly + `cargo install`).
- **No config changes.** No new env vars, no new gateway.toml fields.
