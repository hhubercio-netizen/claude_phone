#### 1.3.4 TOCTOU sweep on `SessionRegistry`

`crates/claude-phone-gateway/src/session/registry.rs:38-66`:

```
register_wrapper:
  44   if self.inner.len() >= self.max_sessions { return Err(...) }   // soft check
  50   let (tx, rx) = mpsc::channel::<Frame>(256);
  51   let session = Arc::new(Session::new(token, tx));
  53   match self.inner.entry(key) {
  54-58    Entry::Occupied(_) => Err(SessionTaken),
  55       Entry::Vacant(v) => { v.insert(session.clone()); Ok(...) }
  65   }
```

The comment at line 41 claims the atomic check happens inside `Entry`, but the
`Vacant` arm at lines 55-59 does **not** re-check `max_sessions`. DashMap holds
per-shard locks, not a global lock, so two concurrent `register_wrapper` calls
with different `token` values that hash to different shards can both:

1. Pass the soft pre-check at line 44 (both observe `len() < max`).
2. Enter different shards' `Vacant` arms simultaneously.
3. Both `v.insert(...)` succeed.
4. `inner.len()` post-condition: `max_sessions + 1` (or more, under heavier
   contention).

The window is small in practice but the bound is documented as a hard cap on
memory consumption (`max_sessions` × per-session buffer = upper bound on RAM).
A few extra sessions over the cap is a memory-DoS amplifier in a fan-out
attacker scenario.

**Fix (in commit 2):** add `AtomicUsize` reservation counter to
`SessionRegistry`. Before allocating any per-session state, `fetch_add(1)`
the counter; if the previous value is `>= max_sessions`, `fetch_sub(1)` and
return error. Decrement on `drop_session`, `remove`, and `sweep_expired`'s
drop path.

```rust
// Sketch — final code in commit 2:
pub struct SessionRegistry {
    inner: Arc<DashMap<String, Arc<Session>>>,
    max_sessions: usize,
    active_count: AtomicUsize, // mirrors inner.len() under a strictly stricter bound
}

pub async fn register_wrapper(&self, token: SessionToken) -> RegisterResult {
    // Atomic reservation (TM-CODE.4): prevents shard-race over-allocation.
    let prev = self.active_count.fetch_add(1, Ordering::SeqCst);
    if prev >= self.max_sessions {
        self.active_count.fetch_sub(1, Ordering::SeqCst);
        return Err(GatewayError::Internal(anyhow::anyhow!("max sessions reached")));
    }

    let key = token.as_str().to_string();
    if key.len() > 64 {
        self.active_count.fetch_sub(1, Ordering::SeqCst);
        return Err(GatewayError::InvalidToken);
    }

    let (tx_to_wrapper, rx_from_phone) = mpsc::channel::<Frame>(256);
    let session = Arc::new(Session::new(token, tx_to_wrapper));

    match self.inner.entry(key) {
        Entry::Occupied(_) => {
            self.active_count.fetch_sub(1, Ordering::SeqCst);
            Err(GatewayError::SessionTaken)
        }
        Entry::Vacant(v) => {
            v.insert(session.clone());
            Ok(WrapperHandle { session, rx: rx_from_phone })
        }
    }
}
```

The token-length check is also moved **before** the channel allocation to
avoid wasting an mpsc per malformed insert (cleanup byproduct; not security
critical).

Regression test in §7: 50 parallel `register_wrapper` calls against a registry
with `max_sessions=5` and 50 *distinct* tokens; assert exactly 5 succeed.

#### 1.3.5 Config bounds sweep

`crates/claude-phone-gateway/src/config.rs:17-20` declares:

```rust
#[serde(default = "default_session_timeout")]
pub session_idle_timeout_secs: u64,    // 7 days default, no bound
#[serde(default = "default_max_sessions")]
pub max_sessions: usize,               // 100 default, no bound
```

A misconfigured TOML could set `session_idle_timeout_secs = u64::MAX`
(`Duration::from_secs(u64::MAX)` is the lifetime of the heat death of the
universe — effectively disables the sweeper) or `max_sessions = usize::MAX`
(allocates a DashMap that can never hit the cap, defeating memory bound).

**Fix (in commit 3):** introduce `GatewayConfig::validate()` that returns
`anyhow::Result<()>` and is called from `GatewayConfig::load` after `toml::from_str`.
Bounds:

| Field                          | Min        | Max                 | Rationale                                                                |
|--------------------------------|------------|---------------------|--------------------------------------------------------------------------|
| `session_idle_timeout_secs`    | 60         | 30 × 24 × 3600      | Below 60 s makes sticky sessions useless; above 30 days no operational use case. |
| `max_sessions`                 | 1          | 10_000              | Below 1 disables service; above 10k each session ≥ 64 KiB → ≥ 640 MiB RAM cap. |

The TM-WS.9 production-vs-public_origin check landed in 4.13 already lives
inside the same `validate()`; this sub-spec adds the two new bounds checks
alongside it.

## 2. Threat-by-threat narrative

### 2.1 TM-CODE.1 — clippy pedantic audit

Existing CI already runs `cargo clippy --workspace --all-targets -- -D warnings`
(see `.github/workflows/ci.yml:20`). This sub-spec adds a one-shot pedantic
pass to the implementation but **does not** add a permanent pedantic gate to
CI. Rationale: pedantic generates noise (e.g., `clippy::module_name_repetitions`,
`clippy::missing_errors_doc`) that does not correlate with security. The
permanent gate stays at `-D warnings` on the default lint set.

The one-shot pass:

1. Run `cargo clippy --workspace --all-targets --all-features -- -W clippy::pedantic` locally.
2. For each warning, classify:
   - **Stylistic noise** (e.g., `module_name_repetitions`, `missing_errors_doc`,
     `must_use_candidate`): ignore.
   - **Real correctness/clarity issue** (e.g., `unnecessary_wraps`,
     `redundant_closure_for_method_calls`, `manual_let_else`): fix inline.
3. Capture the resulting diff in commit 1 of the implementation plan (small,
   stylistic-only).

If the pedantic pass touches a file owned by another sub-spec in flight (e.g.,
files queued for 4.6 or 4.13 changes), the plan agent must coordinate with
those plans — split the commit so 4.14's pedantic-only delta lands cleanly.

### 2.2 TM-CODE.2 — unsafe block isolation

See §1.3.1 for the catalog. The deny-then-allow strategy makes the existing
five unsafe blocks the only unsafe blocks in the workspace, and ensures any
future unsafe is a compile error unless added to `tty.rs` (where reviewers
already pay attention) or explicitly allowed by a reviewer-visible
`#[allow(unsafe_code)]` annotation.

`MaybeUninit::zeroed()` modernization on `tty.rs:11` is the only behaviour
change. The replacement:

```rust
let original: libc::termios = {
    let mut uninit = std::mem::MaybeUninit::<libc::termios>::zeroed();
    // SAFETY: `libc::termios` is plain old data on supported Unix targets
    // (Linux, macOS) — fields are integers, character arrays, and bitflags
    // with no required invariants beyond being syntactically initialized.
    // tcgetattr below overwrites all fields before any read.
    unsafe { uninit.assume_init() }
};
```

is semantically equivalent to the current `std::mem::zeroed()` but explicit
about the assumption being made. Functional behaviour does not change.

### 2.3 TM-CODE.3 — unwrap/expect/panic audit

See §1.3.2 for the catalog. After commit 4 lands, the forward-looking grep
gate `scripts/check_hot_path_unwrap.sh` runs in CI:

```bash
#!/usr/bin/env bash
# scripts/check_hot_path_unwrap.sh
# Fails CI if any production .rs file (i.e. crates/*/src/**) contains a bare
# .unwrap() call, except those tagged with a same-line "// TM-CODE.3: <reason>"
# comment.
set -euo pipefail

bad=$(rg --no-heading -n '\.unwrap\(\)' \
    crates/*/src \
    | grep -v '// TM-CODE\.3:' \
    || true)

if [ -n "$bad" ]; then
    echo "Found .unwrap() in production code without a TM-CODE.3 justification:"
    echo "$bad"
    exit 1
fi
```

Note: `.expect("...")` is allowed without further annotation — the `.expect`
message itself is the documentation. Only bare `.unwrap()` is gated.

The six gateway sites get rewritten to `.expect(JUSTIFICATION)`. No source
file is left with a bare `.unwrap()` in production after this lands; the gate
defends against regression.

### 7.3 New integration test — `tests/registry_race_test.rs`

Single test file owned by this sub-spec. Located at
`crates/claude-phone-gateway/tests/registry_race_test.rs`.

```rust
// crates/claude-phone-gateway/tests/registry_race_test.rs
// Regression test for TM-CODE.4 (shard-race over-allocation against max_sessions).
//
// Spawn N parallel register_wrapper calls with N distinct tokens against a
// registry with max_sessions = K (K < N). Assert that exactly K succeed and
// (N - K) return GatewayError::Internal ("max sessions reached").

#[tokio::test(flavor = "multi_thread", worker_threads = 8)]
async fn registry_enforces_max_sessions_under_concurrent_registration() {
    use std::sync::Arc;
    let registry = Arc::new(SessionRegistry::new(5));

    let mut handles = Vec::new();
    for i in 0..50 {
        let r = registry.clone();
        let token = SessionToken::from_str(&format!("token_{i:020}")).unwrap();
        handles.push(tokio::spawn(async move { r.register_wrapper(token).await }));
    }

    let results: Vec<_> = futures::future::join_all(handles).await;
    let ok_count = results.iter().filter(|r| r.as_ref().unwrap().is_ok()).count();
    assert_eq!(ok_count, 5, "exactly max_sessions=5 must succeed; got {}", ok_count);
    assert_eq!(registry.len(), 5);
}
```

Run requirements:

- `flavor = "multi_thread", worker_threads = 8` — single-threaded runtime
  serializes futures and hides the race. The test must use a multi-thread
  runtime to actually exercise the shard contention.
- 50 distinct tokens against `max_sessions = 5` is enough to produce shard
  contention on DashMap's default shard count. If the plan agent finds the
  test flaky (e.g., never observes >5 successes in the pre-fix code), bump
  the concurrency to 200.

## 9. Backwards compatibility and migration

- **Config-file compatibility:** Commit 3 introduces `validate()` bounds. Any
  existing `gateway.toml` with `session_idle_timeout_secs` between 60 and
  2_592_000 and `max_sessions` between 1 and 10_000 continues to load. The
  current defaults (604_800 and 100) are well inside both ranges — no
  operator-facing migration needed for users on defaults.
- **Operators with non-default configs:** if anyone has set
  `session_idle_timeout_secs = 0` or `max_sessions = 0` (both nonsense),
  the gateway will refuse to start with a loud error. This is the desired
  behaviour — the master spec quality bar disallows silent fixups.
- **Registry behaviour:** Commit 2 changes the failure mode for `register_wrapper`
  under contention from "slightly over the cap" to "exactly at the cap". The
  return type is unchanged (`Err(GatewayError::Internal(...))`); clients see
  the same error envelope they see today when the cap is hit.
- **Test suite:** All existing tests continue to pass without modification.
  New tests are added; no tests are deleted or weakened.
- **CI duration:** `cargo deny check` adds ~30 s to CI total wall-clock. The
  existing `cargo audit` step (similar duration) is kept — total CI growth
  ≈ +30 s. Acceptable.
